────────────────────────────────────────
Overview

• The purpose of curriculum.py is to define an abstract base class—CurriculumScheduler—for curriculum learning strategies. This class provides a common interface so that different methods (such as Self-Paced Learning, teacher-guided scheduling, and loss reweighting methods) can be plugged seamlessly into the training pipeline without modifying the rest of the code. 

• The design follows the provided mermaid class diagram and must exactly adhere to the public “update_schedule(loss: float, epoch: int)” function (and __init__) without adding extra public member functions beyond what is specified in the design.

────────────────────────────────────────
Interface Definition

1. __init__(self, config: dict)
   • Purpose:
     – To initialize the curriculum scheduler using the configuration parameters.
     – To extract curriculum-related hyperparameters from the config dictionary (from the “curriculum” section in config.yaml). For example, keys such as “warmup_epochs”, “schedule_epochs”, and “growth_rate” will be read.
   • Considerations:
     – Since the config values (warmup_epochs, schedule_epochs, and growth_rate) are set to null in the configuration file, the constructor should handle the possibility of missing values. In concrete subclasses, one might define default values or rely on additional external configuration.
     – Store these parameters as internal state (as protected/private attributes) so they can be later used within update_schedule.
     – If further method‐specific parameters are needed (for instance, parameters specifying the range of acceptable loss or thresholds for data rejection) the concrete scheduler can initialize them here.

2. update_schedule(self, loss: float, epoch: int) → float
   • Purpose:
     – To update the internal schedule used for curriculum learning based on the current training loss and epoch.
     – To compute a value (typically a threshold or weight) that will be used by the training loop to decide which data samples to select or how to reweight the loss.
   • Input Parameters:
     – loss: A float representing the training loss (or aggregated loss) at the current training step. This might be used in a self-paced or loss reweighting strategy.
     – epoch: An integer indicating the current epoch number. This allows the method to implement scheduling logic that might depend on the training phase (e.g., warm-up versus later epochs).
   • Expected Output:
     – A float value that represents either a threshold (for data selection) or a scaling factor (for loss reweighting). The exact meaning is left to the concrete scheduler’s implementation; however, the interface always returns a float.
   • Logic to Consider:
     – Warmup Phase: For example, if epoch is less than a configured “warmup_epochs” parameter, the method may simply return a fixed threshold (or unity weight), meaning no curriculum-based adjustment is done in the early phase.
     – Gradual Growth: After the warmup period, the scheduler may adjust the threshold based on a function that could be linear, exponential, or follow another pattern. For example, in self-paced learning, a threshold might increase with epoch, thereby gradually incorporating harder (i.e., higher-loss) samples.
     – Loss Reweighting: For methods that reweight loss, the update_schedule might compute a weight using a function f(loss, epoch) that penalizes high-loss (or noisy) samples during early epochs and gradually relaxes.
     – Teacher Guidance: In teacher-transferring methods (e.g., TTCL), the value returned by update_schedule might reflect a threshold derived by comparing the current loss with a teacher-provided target or guidance metric.
     – It is important to use any math (or PyTorch) functions necessary for implementing these functions (for example, math.exp, math.log, etc.) without deviating from the interface.
   
────────────────────────────────────────
Integration with the Training Loop

• The Trainer (in trainer.py) is expected to invoke update_schedule() at each training step or epoch. The returned float value is then used in one of two typical ways:
   – As a threshold: Data samples with loss values below this threshold might be selected to contribute to the current update.
   – As a weight: The scalar might be used to reweight the loss computed for the mini-batch.
• This mechanism allows the curriculum strategy to be “plugged into” training without altering the overall structure of model forward/backward calls.

────────────────────────────────────────
Pluggable and Extensible Design

• The abstract base class defined in curriculum.py must be designed in a modular fashion so that:
   – New curriculum methods (concrete subclasses) can be added by overriding update_schedule while preserving the same interface.
   – The Trainer need not be modified when switching among curriculum strategies, as it only calls update_schedule(loss, epoch) to get the current scheduling value.
   – For instance, a SelfPacedLearningScheduler concrete class might implement update_schedule() to gradually relax its sample selection threshold based on a predefined growth_rate, while a LossReweightingScheduler might use meta-learning techniques.
• Any additional internal helper functions or attributes should remain encapsulated within the concrete implementations and not alter the public API.

────────────────────────────────────────
Handling Configuration Dependencies

• The scheduler must reference the curriculum-related settings from the config (via config.yaml). No hard-coded values should be assumed; if configuration values (like warmup_epochs) are null, the implementation may either:
   – Set reasonable defaults while documenting the choice, or
   – Defer such initialization logic to the concrete subclass and explicitly log a warning about missing configuration.
• This ensures that adjustments to hyperparameters or scheduling behavior can be made entirely by editing config.yaml without changing code.

────────────────────────────────────────
Summary of Key Points

1. The CurriculumScheduler class serves as an abstract interface:
   – It is constructed with a configuration dictionary that includes hyperparameters from the “curriculum” section.
   – Its key public method, update_schedule(loss: float, epoch: int), computes and returns a float value used for either data selection (threshold) or loss reweighting.
   
2. The implementation must:
   – Support a warmup phase (possibly skipping curriculum adjustments during early epochs).
   – Gradually modify the scheduling parameter based on the current epoch and observed loss.
   – Use mathematical functions and, if necessary, PyTorch functions to implement the scheduling logic.

3. The design is modular, allowing the curriculum learning strategy to be replaced or extended by simply adding new subclasses of CurriculumScheduler that override the update_schedule method.

By adhering to these guidelines, curriculum.py will serve as a robust and reusable component in the overall system for the CurBench benchmark, providing a standardized interface for integrating various curriculum learning methods.
