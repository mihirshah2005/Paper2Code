Overview

• The curriculum.py module establishes an abstract base class—CurriculumScheduler—that defines a standard interface for curriculum learning strategies. Its key method, update_schedule, is intended to return a single float value. In concrete implementations, however, this value can represent different quantities (e.g., a sample selection threshold in Self-Paced Learning (SPL) or a weight factor in loss reweighting).  
• It is critical that concrete subclasses clearly document and separate out their internal logic so that there is no ambiguity when the Trainer applies the returned value.

__init__(self, config: dict)

• Purpose:
  – Initialize the scheduler by extracting curriculum-related hyperparameters from the “curriculum” section of the configuration file.
  – Parameters typically include warmup_epochs, schedule_epochs, and—if needed for a specific strategy—an initial_threshold and growth_rate.  
• Considerations:
  – Since the base configuration does not explicitly define parameters such as initial_threshold, concrete subclasses that require these values must either:
     • Extend the configuration by documenting the new parameter requirements, or  
     • Define a default internally and clearly document that choice.
  – The __init__ method stores these parameters as part of the internal state without assuming a single computation logic; the interpretation is left to each concrete strategy.

────────────────────────────────────────
update_schedule(self, loss: float, epoch: int) → float

• Purpose:
  – To compute and return a float value that will drive the curriculum mechanism.  
  – Depending on the strategy, this value could serve as one of the following:
      • A data selection threshold (for SPL), meaning that only samples with loss below this threshold are chosen for training.
      • A weight factor (for loss reweighting), which scales the contribution of the computed loss.
      • A value derived from a comparison between teacher and student loss (for teacher-guided scheduling).

• Addressing Specific Critiques:

  1. Teacher-Guided Scheduling:
     – Critique: The signature (loss, epoch) does not allow direct incorporation of a teacher’s guidance metric.
     – Resolution: Although the abstract method’s parameters are limited, concrete implementations for teacher-guided scheduling can adopt one of these approaches:
         • Extend the method signature using optional keyword arguments (for example, defining update_schedule(self, loss: float, epoch: int, **kwargs) → float) so that a teacher_loss (or guidance metric) can be passed in when available.
         • Alternatively, a teacher-guided scheduler may store the teacher’s loss as an instance attribute (set externally via initialization or a setter method) before calling update_schedule.
         
     – In either case, the concrete subclass must clearly document how it uses the teacher’s metric in the scheduling logic.

  2. Ambiguous Interpretation of Return Value:
     – Critique: The returned float value is used for different purposes (threshold, weight), leading to ambiguity.
     – Resolution: Each concrete subclass must explicitly document the meaning of the returned float. For instance:
         • A Self-Paced Learning scheduler would state, “This float represents the current selection threshold. If a sample’s loss is below this threshold, it is selected for training.”
         • In contrast, a Loss Reweighting scheduler would document, “The returned float is a multiplicative weight used to scale the loss of the sample.”
     – The Trainer must be designed (or configured) to interpret the float correctly based on the chosen curriculum strategy.

• General Internal Logic Guidelines:
  – For Self-Paced Learning (SPL):
      • If the current epoch is within a warmup period (epoch < warmup_epochs), return a baseline threshold (e.g., 1.0 or a configured value).  
      • Once past warmup, update the threshold using a clear mathematical function such as a linear or exponential growth based on (epoch – warmup_epochs) and the configured growth_rate.
      • If an “initial_threshold” is needed for SPL, the concrete subclass must either require its presence in the configuration or set and document a default value.
      
  – For Teacher-Guided Scheduling:
      • Retrieve the teacher’s loss (either via an additional parameter through **kwargs or from an instance attribute).
      • Compute a scheduling value that compares the student’s current loss with the teacher’s loss. For example, the scheduler might return the relative difference or a dynamically computed threshold to guide data selection.
      
  – For Loss Reweighting Methods:
      • Compute and return a weight factor using the current loss. This computation may involve nonlinear functions (e.g., inverse functions or a meta network output) that attenuate the influence of samples with high, potentially noisy, loss.
      • The concrete subclass must describe how this factor evolves through training (for example, being more aggressive during early epochs and gradually relaxing).

Integration with the Training Loop

• The Trainer (in trainer.py) will call update_schedule(loss, epoch) (or update_schedule(loss, epoch, teacher_loss=...) if keyword arguments are supported) to obtain the current scheduling value.
• It is the responsibility of the concrete scheduler to ensure that the value and its meaning (threshold vs. weight) are documented and consistently used within the Trainer without any guesswork.

Summary

• The CurriculumScheduler abstract class provides a minimal interface with __init__ and update_schedule that all curriculum learning strategies must follow.
• While the update_schedule method’s signature accepts only loss and epoch, concrete implementations (especially for teacher-guided scheduling) must use mechanisms—such as optional keyword arguments or pre-stored teacher metrics—to incorporate teacher guidance.
• The returned float value may represent a threshold for data selection or a weight for reweighting loss; concrete subclasses must clearly document its interpretation to avoid ambiguity in how the Trainer applies it.
• Parameters like “initial_threshold” must be explicitly defined or documented in concrete implementations to prevent misconfiguration; if not provided in the configuration file, well-documented defaults should be used.

By following these guidelines, the abstract curriculum.py module will provide a robust, extendable, and unambiguous interface for various curriculum learning strategies while clearly delineating the implementation details required for each specific method.
