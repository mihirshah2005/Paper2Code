Below is a detailed logic analysis for writing the code in model.py. This module is responsible for instantiating backbone models for different domains (CV, NLP, Graph) based on the configuration file and dataset properties. The ModelManager class will serve as a unified interface for creating the proper model instance, adjusting input/output layers as needed, and handling pre‐trained weight loading when applicable.

────────────────────────────
1. Overall Structure

• The main class will be ModelManager. In its __init__ method, it will accept a configuration dictionary (loaded from config.yaml via config.py) so that the same hyperparameters and model identifiers are used across the experiment.

• ModelManager will provide a public method—build_model—that takes parameters such as:
  – domain (e.g., "cv", "nlp", "graph")
  – model_name (e.g., "LeNet", "ResNet-18", "ViT" for CV; "LSTM", "BERT", "GPT2" for NLP; "GCN", "GAT", "GIN" for Graph)
  – num_classes: the number of output classes (obtained from the dataset loader)
  – additional dataset-specific information (e.g., input dimensions for images, sequence lengths, or feature dimensions for graphs).

• The design may include helper methods such as build_cv_model(), build_nlp_model(), and build_graph_model() that are internally called by build_model to handle domain-specific instantiation logic.

────────────────────────────
2. Domain-Specific Model Instantiation

A. CV Models
 • LeNet:
  – Implement a custom LeNet architecture using nn.Conv2d layers and nn.Linear layers.
  – Ensure that the final classification (or fc) layer has an output size equal to num_classes.
 • ResNet-18:
  – Use torchvision’s resnet18 (with or without pretrained weights as per experiment design).
  – After instantiation, replace or modify the final fully connected layer (commonly “model.fc”) to nn.Linear(in_features, num_classes).
 • ViT:
  – Either use a Vision Transformer implementation from torchvision or a custom implementation.
  – Adjust the classification head (or output projection) to output num_classes.

B. NLP Models
 • LSTM:
  – Build a custom LSTM-based classifier.
  – Likely include an embedding layer (if inputs are token indices), an nn.LSTM module, and a subsequent fully connected layer to produce logits for num_classes.
 • BERT:
  – Leverage Hugging Face’s Transformers library.
  – Use BertForSequenceClassification (e.g., from_pretrained("bert-base-uncased", num_labels=num_classes)), which automatically adjusts the classification head.
 • GPT2:
  – Use GPT2ForSequenceClassification from the Transformers library (e.g., from_pretrained("gpt2", num_labels=num_classes)).
  – Note that GPT2 is originally designed for language modeling, so ensure that any necessary modifications (such as padding or adding a classification head) are applied.

C. Graph Models
 • GCN:
  – Construct a model using torch_geometric.nn modules, for example stacking one or multiple GCNConv layers.
  – After convolutional layers, add a global pooling (e.g., global_mean_pool) and a final linear layer sized for num_classes.
 • GAT:
  – Similar to GCN, but use GATConv layers.
  – Ensure that the attention-based mechanism is properly configured, and follow with pooling and a classification head.
 • GIN:
  – Use GINConv layers (possibly with a simple Multi-Layer Perceptron as the update function) following ideas from Weisfeiler-Lehman tests.
  – Again, include a final classifier adjusted to num_classes.

────────────────────────────
3. Configuration and Adjustments

• The ModelManager must refer to the settings in config.yaml under the “training” section to determine which models are supported per domain.

• When a model is instantiated the following adjustments should be made:
  – For CV models:
   ○ Adjust the first layer if input channels differ (for example, if using grayscale images versus RGB).
   ○ Replace the final classification layer with one that outputs num_classes.
  – For NLP models:
   ○ Set the “num_labels” parameter in the classification head.
   ○ If using LSTM, possibly include parameters for hidden size based on external configuration.
  – For Graph models:
   ○ Set the number of node features and the output dimension appropriately based on the dataset.
   ○ Configure the number of convolutional layers and hidden units as required (these can be based on default fixed values or additional config parameters).

• Pretrained weight loading is essential for BERT and GPT2:
  – Call from_pretrained() to load the models with standard pretrained parameters (e.g., "bert-base-uncased" for BERT or "gpt2" for GPT2).
  – Ensure that the classifier head is adjusted to have num_classes outputs.

────────────────────────────
4. Integration with the Training Pipeline

• The returned model must be a PyTorch nn.Module such that the Trainer (in trainer.py) can directly invoke the forward(x) method during training and evaluation.

• Regardless of the internal structure (custom models for LeNet/LSTM or API-based models for Transformer and graph networks), the interface remains consistent.

• The ModelManager might also include logging (or at least comments) about model summary, parameter count, etc., which can be useful later when comparing reported complexity (time, memory) from the experiments.

────────────────────────────
5. Error Handling and Flexibility

• Include error messages or assertions if an unsupported model name or domain is provided.

• Document any assumptions, for instance:
  – The input dimensions for CV models should be passed via dataset information.
  – For NLP models, tokenization and maximum sequence length are handled elsewhere (in the dataset loader).

• Keep the design modular so that new model types or modifications (like additional backbone models) can be added later without changing the common interface.

────────────────────────────
6. Dependencies and Libraries

• Import necessary PyTorch modules (torch, torch.nn, torchvision.models).

• For NLP models, import the Transformers library (from transformers import BertForSequenceClassification, GPT2ForSequenceClassification).

• For Graph models, import from torch_geometric.nn (e.g., GCNConv, GATConv, GINConv) or leverage any available base classes.

Summary

The logic for model.py centers on creating a ModelManager class that reads the experiment configuration, determines the correct model type based on domain and model identifier, instantiates the model with proper adjustments for input/output dimensions, and returns a usable nn.Module for training and evaluation. By abstracting the construction of CV, NLP, and Graph models into domain-specific helper functions, we achieve a modular and extendible design that conforms to the specifications in CurBench. Future modifications (such as changing hyperparameters for new curriculum learning experiments) will only require updates in configuration or small adjustments in this module.

This analysis provides a clear blueprint for writing model.py that integrates seamlessly with the rest of the system as outlined in the experimental plan and configuration file.
