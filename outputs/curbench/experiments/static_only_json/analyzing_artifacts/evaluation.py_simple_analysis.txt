1. Class Initialization

• The Evaluation class’s __init__ method accepts three parameters:
 – model: The trained PyTorch model (or the backbone wrapped with necessary post‐processing) to be evaluated.
 – data: A dictionary containing the evaluation data loaders. This dictionary is expected to include keys for “train”, “val”, and/or “test”. For our benchmarking, note that:
  – CV and Graph experiments use the test set (if available) or fall back to the validation set.
  – NLP experiments typically use the validation set (because test labels are not provided).
 – config: The experiment configuration dictionary (parsed from config.yaml) that holds evaluation metric settings and other relevant parameters.

• Domain Determination:
 – The evaluation logic needs to know the target domain (CV, NLP, or Graph) because the metrics differ.
 – One option is to include a “domain” key inside the data dictionary (or deduce it from config if the experiment is organized by domain under config.training) so that Evaluation can decide which metric(s) to compute.
 – For example, config["evaluation"]["metrics"] is given as:
  • cv: "accuracy"
  • nlp: ["accuracy", "F1", "Spearman", "Matthews"]
  • graph: "AUC"
 – The Evaluation class uses this mapping to compute domain-specific metrics.

2. The evaluate() Method

The evaluate() method is responsible for:

A. Model Preparation and Data Selection

 – Set the model to evaluation mode (model.eval()).
 – Decide on which data loader to iterate over. For example, if a “test” loader is available in the data dictionary, use that; otherwise, use “val.”
 – Initialize empty containers (e.g., Python lists) to collect model outputs (predictions) and corresponding ground-truth labels.

B. Iterating the Data Loader (Inference Loop)

 – Use torch.no_grad() to disable gradient computation during evaluation.
 – For each batch:
  • Unpack the inputs and targets. The exact unpacking may vary by domain:
   – CV: inputs will be image tensors and targets as integer class labels.
   – NLP: inputs are tokenized sequences (or input IDs/attention masks) and targets are labels.
   – Graph: inputs may be PyG Batch objects containing node features, edge indices, etc., and targets are class labels (often binary for ogbg-molhiv).
  • Call the model’s forward pass (e.g., outputs = model(inputs)).
  • Post-process the outputs:
   – For CV and NLP tasks, if the model outputs logits (of shape [batch, num_classes]), apply argmax(dim=1) to obtain the predicted class.
   – For graph tasks (AUC evaluation), if the task is binary classification, convert the logits to probabilities using a sigmoid (or softmax for multi-class) so the continuous score can be used for computing AUC.
  • Append the predictions and the ground truth labels to their respective lists.
   – Ensure that the predictions and targets are moved to CPU and converted to numpy arrays (since scikit-learn functions require NumPy arrays).

C. Metric Computation

 1. After processing the entire dataset, merge the collected predictions and targets into NumPy arrays.
 2. Based on the target domain (determined via the data dictionary or config), compute the relevant metrics:
  – For CV:
   • Use sklearn.metrics.accuracy_score to compute the accuracy.
  – For NLP:
   • Compute accuracy with accuracy_score.
   • Compute F1 score using f1_score (typically with average="macro" to account for multi-class tasks).
   • Compute Matthews correlation coefficient via matthews_corrcoef.
   • Compute Spearman correlation:
    – Since Spearman is not directly provided by scikit-learn, you can implement it by ranking both the predictions and ground truth (for example, using NumPy’s argsort) and then computing the Pearson correlation of the ranked arrays. (If allowed, you may also use scipy.stats.spearmanr; however, note that no extra third-party packages are allowed beyond what is specified.)
  – For Graph:
   • For binary classification tasks (e.g., ogbg-molhiv), use sklearn.metrics.roc_auc_score with the predicted probabilities (and the ground-truth labels) to compute the AUC.
 3. The computed metric(s) for the given domain are then stored in a dictionary. For instance:
  • If the domain is “cv” → result_dict = {"accuracy": computed_value}
  • If the domain is “nlp” → result_dict = {"accuracy": value, "F1": value, "Spearman": value, "Matthews": value}
  • If the domain is “graph” → result_dict = {"AUC": computed_value}

D. Returning the Summary Dictionary

 – Finally, the evaluate() method returns the summary dictionary containing all the computed metrics. This dictionary will be used later to log the metrics and compare baseline versus curriculum learning results in tables (similar to Table 3 in the paper).

3. Additional Considerations

• Data Format and Domain-Specific Preprocessing:
 – Ensure that the Evaluation module is compatible with the data formats returned by the DatasetLoader. For example, if the loader attaches an extra “domain” flag or if the batch structure differs per modality, include checks or helper functions to correctly unpack the batch.
 – Document any assumptions regarding the order of inputs and the shape/format of labels.

• Handling Continuous vs. Discrete Outputs:
 – For classification tasks (CV and NLP), convert raw model outputs to discrete class predictions using argmax.
 – For AUC on graph datasets (or other binary tasks), retain continuous output probabilities by applying a sigmoid function if the model does not already output probabilities.

• Reproducibility:
 – Although complexity metrics (training time, GPU memory) are recorded during training, make sure the evaluation does not inadvertently modify the state (e.g., set torch.no_grad() properly) to ensure reproducibility.
 – The evaluation results for each run (across the fixed seeds provided in config.yaml) should be logged externally so that averages and standard deviations can be computed over multiple runs.

• Use of scikit-learn Functions:
 – accuracy_score, f1_score, and matthews_corrcoef from scikit-learn are used directly.
 – For roc_auc_score, ensure that the true labels and predicted probabilities are in the appropriate format (e.g., if the labels are binary, the predicted output should be a single probability per sample).
 – For Spearman correlation, implement a helper function (or use an allowed third-party if justified) that first ranks the prediction and target arrays and then computes the Pearson correlation coefficient of these ranked arrays.

4. Summary of Evaluation Class Workflow

1. Initialize:
 • Store model, evaluation data loader(s), and configuration.
 • Determine the domain (CV, NLP, or Graph) based on passed data/config parameters.

2. Evaluate:
 • Set model mode and iterate over the selected data loader.
 • For each batch, perform inference (in a torch.no_grad() block), post‐process outputs into discrete class predictions or continuous probabilities as needed, and accumulate the results.
 • After processing all batches, compute the required evaluation metrics based on the domain using scikit-learn functions (and a custom ranking function for Spearman).
 • Build and return a summary dictionary of the evaluation metrics.

3. Output:
 • The returned dictionary will contain key-value pairs mapping the metric names (e.g., “accuracy”, “F1”, “Spearman”, “Matthews”, “AUC”) to their computed values, which will later be logged or aggregated across multiple runs.
