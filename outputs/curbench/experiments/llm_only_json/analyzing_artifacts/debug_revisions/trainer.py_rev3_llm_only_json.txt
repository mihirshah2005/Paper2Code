────────────────────────────
• Inputs:
  – model: Backbone network instance (from ModelManager) expected to provide a forward() method.
  – data: Dictionary with training, validation, and test datasets (from DatasetLoader).
  – curriculum: Instance of a CurriculumScheduler (or its subclass) implementing update_schedule().
  – config: Configuration dictionary loaded from config.yaml containing domain-specific hyperparameters and settings.

  – The Trainer now requires an explicit domain specification. It will check config for a key (e.g., config["domain"]) or use an attribute provided by model metadata.
  – If neither is available, rather than defaulting ambiguously, the Trainer raises a ValueError with a clear message such as: "Domain information missing. Please specify the training domain (e.g., 'cv', 'nlp', or 'graph') in the configuration."
  – This eliminates misconfiguration of loss functions and optimizer setups.

• Initialization Steps:
  – Once the domain is determined (e.g., "cv", "nlp", or "graph"), the Trainer instantiates:
   ○ The appropriate loss function (for example, CrossEntropyLoss for classification tasks).
   ○ The optimizer, choosing among Adam, AdamW, or SGD based on the domain and config values.
  – It validates external dependencies:
   ○ Checks that model implements a callable forward() method.
   ○ Verifies that curriculum implements update_schedule(loss: float, epoch: int).
   ○ Confirms that the data dictionary contains the required DataLoader objects.
  – Establishes logging utilities and, if available, sets up GPU memory tracking (conditional on torch.cuda.is_available()).

2. Training Loop (train() Method)
────────────────────────────
The train() method now incorporates a more modular integration for curriculum updates and refined error handling.

A. Epoch-Level Loop:
  – Record overall training start time using time.time().
  – Loop over epochs as specified by config.
  – At the epoch’s start, reset or log epoch-specific metrics and (if on GPU) reset GPU memory counters.

B. Mini-Batch Loop:
For each mini-batch from the training DataLoader, the following steps are executed with dedicated error handling blocks:

  1. Data Preparation:
   – Move the input data and labels to the target device. (Before using GPU-specific functions, check with torch.cuda.is_available() to avoid issues in CPU-only environments.)

  2. Model Forward Pass:
   – Wrap the forward pass in a dedicated try/except block.  
   – If a ForwardError (or Exception) is caught, log detailed information (batch index, input shape, error type) and decide whether to skip the batch or gracefully abort training.

  3. Loss Computation:
   – Compute the raw loss using the selected loss function.

  4. Curriculum Scheduler Integration:
   – Instead of a simple loss scaling, the Trainer now notes that curriculum strategies might be more advanced.
   – Call curriculum.update_schedule(raw_loss, epoch) within its own try/except block.
    • The update_schedule method is expected to return either a scaling factor or, in an advanced mode (if the config specifies a “curriculum_mode”), a flag or even an updated batch ordering.
    • In this initial implementation, the returned value is used as a multiplier on the loss. Future extensions may integrate data reordering or dynamic sampling prior to the forward pass.
   – Any errors specific to the curriculum update are logged separately with context (e.g., current loss and epoch).

  5. Backpropagation and Optimizer Step:
   – Use separate try/except blocks for:
    a. Clearing gradients and executing loss.backward(), capturing errors related to gradient computation.
    b. Calling optimizer.step(), capturing and logging optimizer-specific errors.
   – In each block, log the exception type and message to facilitate debugging.

  6. Batch-Level Logging:
   – Accumulate and log mini-batch loss values.
   – If running on GPU (torch.cuda.is_available() returns True), log torch.cuda.max_memory_allocated() as the current memory usage. In CPU-only environments, skip or log “GPU memory not available.”
   – Log additional details, such as the obtained curriculum scaling factor (or other scheduling information) for each batch.

C. End-of-Epoch Processing:
  – After all mini-batches, log the epoch’s elapsed time.
  – Optionally perform a validation pass to monitor performance.
  – Record and log the epoch-level metrics (average loss, learning rate adjustments, peak GPU memory if applicable).
  – Update any learning rate schedulers or curriculum parameters if required.

D. Finalization:
  – Record overall training completion time and other global complexity metrics.
  – Return the final trained model and training statistics for downstream evaluation.

3. Curriculum Scheduler Integration
────────────────────────────
• The Trainer now documents that the curriculum scheduler integration—via update_schedule(loss, epoch)—is intentionally designed to support both simple loss scaling and more advanced strategies.
  – In “simple” mode, update_schedule returns a scaling factor that multiplies the loss.
  – In “advanced” mode (should the configuration allow and the scheduler support it), update_schedule may return directives for sample reordering or reshuffling for dynamic sampling.
  – The Trainer logs the curriculum output separately and flags if the returned value differs from a scalar weight.
  – This design is adaptive; however, if only a scalar is returned, the Trainer performs the usual loss scaling.

────────────────────────────
4. GPU Memory Usage Logging
────────────────────────────
• Before invoking torch.cuda.max_memory_allocated(), the Trainer checks torch.cuda.is_available() to ensure training is on a GPU.
  – If not, the Trainer logs that GPU memory statistics are not applicable or available.
  – This prevents errors and misleading log entries in CPU-only environments.

5. Integration and External Component Validation
────────────────────────────
• The Trainer explicitly documents and validates the required interfaces for its external dependencies:
  – ModelManager: Model must implement a callable forward() method.
  – DatasetLoader: Data should be provided in a dictionary containing valid DataLoader objects for training (and optionally validation/test).
  – CurriculumScheduler: Must implement update_schedule(loss: float, epoch: int) and return a scalar (or advanced directive) that can be applied to the loss.
• The Trainer performs runtime checks (using hasattr() and callable()) where applicable, and raises descriptive exceptions if any component does not conform to these interfaces.
  – This ensures that if actual implementations differ, the Trainer will catch integration issues early and provide clear guidance for correction.
