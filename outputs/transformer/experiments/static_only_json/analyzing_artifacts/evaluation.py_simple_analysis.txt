1. Initialization and Setup

• Evaluation Class Constructor:
  – The Evaluation class will be instantiated with three primary inputs: the trained TransformerModel instance, the test DataLoader (i.e., the dataset of examples to evaluate), and a configuration dictionary parsed from the YAML file.
  – In __init__(…), the class should extract the inference settings (e.g., beam_size, length_penalty, max_length_offset) from the config. Note that these settings differ for translation and parsing:
    • For translation: 
         ○ beam_size = config["inference"]["translation"]["beam_size"] (expected value 4)
         ○ length_penalty = config["inference"]["translation"]["length_penalty"] (expected value 0.6)
         ○ max_length_offset = config["inference"]["translation"]["max_length_offset"] (50)
    • For parsing:
         ○ beam_size = config["inference"]["parsing"]["beam_size"] (expected value 21)
         ○ length_penalty = config["inference"]["parsing"]["length_penalty"] (expected value 0.3)
         ○ max_length_offset = config["inference"]["parsing"]["max_length_offset"] (300)
  – The Evaluation instance should also store a flag indicating which type of task (translation or parsing) is being run. This decision might be made indirectly by inspecting the structure of config["data"] or by a separate parameter.

• Preparation for Inference:
  – Before starting evaluation, the model must be switched to evaluation mode (model.eval()).
  – Use torch.no_grad() to disable gradient tracking so that the inference loop is fast and memory efficient.

2. Beam Search Decoding Process

• Decoding Overview:
  – The primary inference method (e.g., evaluate()) will loop over the test DataLoader. For each test sample (or batch), the goal is to generate output sequences using beam search.
  – Beam search is used to expand multiple candidate sequences at each time step. At each decoding step:
    • Work from the current beam candidates.
    • Use the model’s forward() method to get probability distributions over the vocabulary for the next token.
    • Apply the appropriate masking (the decoder is expected to include its own mask to enforce left-to-right generation).
  
• Beam Search Parameters:
  – Use the beam_size and length_penalty values from the config.
  – For each input sequence, the maximum allowed output length is computed as (input_length + max_length_offset).
  – The length penalty adjusts scores by, for example, dividing (or otherwise scaling) the log probability by (length)^α (as defined in the paper). Implement this adjustment when scoring complete candidate sequences.
  
• Beam Search Implementation Detail:
  – Develop (or include as a helper method) a _beam_search() function that performs the following:
    1. Initialize the beam with the start-of-sequence token.
    2. For each time step, for every candidate in the beam, feed the current decoder input (and associated mask) to the model’s forward() function.
    3. Obtain output logits, apply softmax to obtain probabilities.
    4. Expand each candidate with the top k tokens based on probability, where k is the current beam size.
    5. Compute the adjusted score for each new candidate using the log probability and the length penalty.
    6. Prune the beam so that only the top beam_size candidates survive.
    7. Implement early stopping by checking if all candidates have generated the EOS token or if maximum length is reached.
  – Ensure that for each batch sample, this decoding is done independently (or vectorized if possible).

3. Metric Computation and Aggregation

• Collecting Outputs:
  – As predictions are generated for each input in the test set, collect both the candidate output sequences and the corresponding ground-truth (reference) sequences.
  – Depending on the task:
    • For machine translation, the evaluation metric is the BLEU score.
    • For constituency parsing, the metric is usually the F1 score with respect to the gold parse trees.
  
• Computing BLEU Score:
  – Once all candidate and reference sequences have been collected, compute the BLEU score. This can be done using an existing library (e.g., NLTK’s corpus_bleu function) if available.
  – Make sure that tokenization of outputs is consistent with the evaluation (e.g., remember that input and output embeddings were created using SentencePiece with the given vocabulary size).
  
• Computing F1 Score for Parsing:
  – If evaluating a parsing task, the output produced by the model must be post-processed (if needed) into a format that can be compared with the gold parse trees.
  – Use an established evaluation method or script to compute labeled/unlabeled F1 scores.
  
• Return and Logging:
  – The evaluate() method should return a dictionary of evaluation metrics (e.g., {"BLEU": <value>}) or ({"F1": <value>}) and also print/log a summary.
  – The evaluation process should also log progress (for example, displaying evaluation progress per batch or percentage completed).

─────────────────────────────  
4. Integration with the Rest of the Code

• Input and Output Interface:
  – The Evaluation class uses the forward() function from the TransformerModel to generate predictions. The model instance is passed from Main (see the program call flow).
  – Ensure that the Evaluation class does not add any public member functions that do not already exist per the design. Only the evaluate() method (and possibly private helper methods like _beam_search) should be added.
  
• Configuration and Reproducibility:
  – All inference hyperparameters (beam size, length penalty, maximum output length offset) must be read directly from the config file (config.yaml) as shown above.
  – No assumptions should be made about default values beyond what is specified in the YAML file.
  – The Evaluation class should also be flexible enough to handle evaluations for both translation and parsing tasks, if necessary, based on the config.

─────────────────────────────  
5. Additional Considerations

• Masking in Beam Search:
  – Ensure that at every decoding step the proper mask is applied so that the decoder cannot “peek” at future positions. This is crucial for maintaining the autoregressive nature of the Transformer as described in the paper.
  
• Efficiency:
  – Optimize the beam search loop to ensure that candidate expansion and pruning are performed efficiently, especially since long sequences and complex models may result in high computational costs during inference.
  – Use batching where possible when processing multiple beam candidates (taking care to correctly update and manage masks for each candidate sequence).
  
• Logging and Debugging:
  – Provide detailed logging of predictions and their associated scores so that the behavior of individual attention operations (if required later) can be investigated.
  – Optionally, include a “verbose” mode that can output individual beam search steps, without affecting performance during standard evaluation runs.
