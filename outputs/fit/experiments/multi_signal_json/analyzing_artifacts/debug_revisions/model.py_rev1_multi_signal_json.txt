1. Overall Structure and Initialization

• The Model class is initialized with a parameters dictionary (extracted from the configuration file). In __init__, the following key setup steps must be performed:
  – Read model-specific parameters (e.g., patch_size = 2, max_token_length = 256) and transformer settings (positional_embedding set to “2D RoPE”, attention set to “Masked MHSA”, and ffn set to “SwiGLU”).
  – Load and store the pretrained VAE using the path from config ("huggingface/stabilityai/sd-vae-ft-ema"). This is done in a dedicated load_pretrained_vae(vae_path: str) method.
  – Initialize any necessary linear projection layers for token embedding (if required by design) and define a “patchify” function to convert VAE latent maps into a sequence of tokens.
  – Construct the transformer blocks. Each block should mimic DiT’s architecture (number of layers, hidden size, and heads as defined in DiT-B/2 or DiT-XL/2) but with these modifications:
      ▪ Replace the standard MHSA with a Masked MHSA implementation (see Section 3.3 of the paper).
      ▪ Replace the standard MLP in the FFN with a SwiGLU block.
  – Set up the positional embedding computation module to use 2D RoPE. This function will compute positional embeddings for every patch—by separately computing 1D RoPE embeddings for the height and width coordinates and then concatenating them.

──────────────────────────────
2. Pretrained VAE Loading and Interfacing

• Method: load_pretrained_vae(vae_path: str)
  – This method loads the off‐the‐shelf VAE (from Hugging Face) using the provided checkpoint path.
  – Once loaded, the VAE is stored in an instance variable (e.g., self.vae) so that later in a forward pass the image can be encoded into latent codes before tokenization.
  
• Additional helper methods may include separate encode() and decode() wrappers around the VAE to clearly distinguish between synthesizing latent tokens and decoding them into images.

3. Tokenization: Patchifying VAE Latent Output

• Process:
  – After encoding an input image with the VAE encoder, the latent output (with dimensions [Batch, C, H_latent, W_latent]) is “patchified.”
  – Using the patch size p = 2 (from the config), segments of the latent map are extracted. For example, using an unfolding operation (or reshaping) to convert the latent feature map into a sequence.
  – Calculate the natural token sequence length L (which equals the number of patches) for each sample.
  – Each sequence is then padded to the fixed maximum token length (L_max = 256, as specified in config). The function should also produce a corresponding token mask that marks “real” tokens (valid patches) and padded tokens.
  – This mask is later used by the Masked MHSA to ensure that no padded tokens contribute to attention computations (typically by assigning a very negative value, e.g., –∞).

──────────────────────────────
4. Positional Embedding via 2D RoPE

• Key Points:
  – Instead of using learnable absolute positional embeddings, compute the positional embeddings based on 2D Rotary Positional Embedding (RoPE).
  – For each token (which has an implicit 2D coordinate from the patch grid), compute separate 1D RoPE embeddings:
      ▪ One for the height coordinate.
      ▪ One for the width coordinate.
  – These 1D embeddings are then concatenated along the last dimension to form the full 2D RoPE embedding.
  – Use the fixed rotary base (b = 10000) and follow the formulas given in the paper. In particular, if in the inference phase the resolution exceeds that of the training distribution, adjust the scaling factor s = max(max(H_test, W_test) / L_train, 1.0) where L_train is derived from L_max (e.g., L_train = √256).
  – Additionally, the design should allow for switching between the vanilla 2D RoPE and a decoupled version (where height and width rotary frequencies are computed separately) if needed for improved extrapolation.
  – Implement this logic in a dedicated helper method (for example, get_2d_rope_positional_embedding), which takes in token spatial coordinates and the optional scale factor as arguments.

──────────────────────────────
5. Building the FiT Transformer Blocks

• Each transformer block must integrate the following modifications:
  – Masked MHSA:
      ▪ Implement a multi-head self-attention mechanism that accepts an additional attention mask. This mask is created from the tokenization step and indicates which tokens are “real” (mask value 0) and which are padded (mask value –inf).
      ▪ The softmax on the attention logits should use this mask so that padded tokens do not contribute to the computation.
  – SwiGLU FFN:
      ▪ Replace the traditional MLP block with the SwiGLU block. The implementation should follow the formula: f(x) = (SiLU(W₁x) ⊗ W₂x) followed by a linear projection (W₃x).
      ▪ This module should be built as part of each transformer block.
  – Standard operations:
      ▪ Include layer normalization and residual connections on both the attention and FFN sublayers.
  – The collection of transformer blocks is then applied sequentially on the token embeddings (combined with the 2D RoPE positional embeddings).

6. Positional Embedding Interpolation Methods for Resolution Extrapolation

• The model must support several positional interpolation methods:
  – Baseline methods: PI (Position Interpolation) and EI (Embedding Interpolation) for handling resolution differences.
  – Training‐free methods adapted from LLM literature: NTK and YaRN.
  – Their vision‐specific adaptations: VisionNTK and VisionYaRN.
  
• Implementation:
  – Create dedicated helper functions (or a unified method with a “method” parameter) that, given a computed positional embedding and a target resolution, adjust the embedding using the appropriate formula.
  – These functions must use the same hyperparameters as in the paper (for instance, for YaRN, use α = 1 and β = 32) and the scale factor logic from config.
  – When the aspect ratio is not 1:1, the decoupled calculations (separately for height and width) should be applied.
  – During inference, before adding the positional embeddings to the token embeddings, pass them through the selected interpolation method (the method to use can be chosen based on an additional parameter or experimental setting).

──────────────────────────────
7. Forward Pass and Integration with Diffusion

• The forward(x: Tensor) method follows these steps:
  1. Input image x is passed to the pretrained VAE encoder to obtain latent representations.
  2. Apply the patchify function on the latent map:
      – Convert the latent map into a sequence of tokens.
      – Pad the sequence to max_token_length = 256 and generate the corresponding token mask.
  3. Optionally project the patches into an embedding space if required.
  4. Compute 2D RoPE positional embeddings for each token based on its spatial (patch grid) coordinates.
      – If resolution extrapolation is needed (e.g., when working with out-of-distribution resolutions), apply the chosen interpolation method (e.g., VisionNTK or VisionYaRN) to the positional embeddings.
  5. Combine (typically add) the token embeddings with the positional embeddings.
  6. Pass the combined embeddings through the FiT transformer blocks:
      – In each block, use the attention mask in the Masked MHSA to properly ignore padded tokens.
      – Use the SwiGLU feed-forward module within each block.
      – Ensure residual connections and layer normalization are applied as standard.
  7. The output is a sequence of denoised token representations that directly relate to the latent space used in the diffusion process.
  8. (If applicable) A subsequent step can “unpatchify” these tokens and forward them to the VAE decoder for image reconstruction. However, depending on how the trainer integrates the diffusion model, the forward method might only be responsible for producing the transformer output.

• Note on Diffusion Details:
  – Although the diffusion noise schedule and timestep conditioning are specified in the trainer, the model should support receiving any extra conditioning signals (like diffusion timestep embeddings) if needed.
  – Make sure that any additional inputs (e.g., class labels or time-step embeddings for the diffusion process) are appropriately integrated with the transformer input.

• Ensure that all operations (patchify, positional embedding, masking, transformer computations) consistently respect device placement and data types.
• The model’s output, along with the token mask, must allow the Trainer to compute the loss only over “real” tokens (non-padding). This means that the forward method should either output or internally keep track of the mask.
