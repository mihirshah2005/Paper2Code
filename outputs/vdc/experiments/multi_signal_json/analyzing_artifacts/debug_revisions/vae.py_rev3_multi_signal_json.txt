1. Overview and Naming Clarification

• Purpose:  
  – This module implements the Visual Answer Evaluation (VAE) component of the VDC pipeline. It is important to note that although the module is named "VAE," it does not refer to a variational autoencoder. Instead, it performs binary evaluation of answers (i.e., determining whether the provided answer agrees with the expected semantics) as described in the paper.  
  – This clarification should be included in the module-level documentation to prevent confusion with models that have latent variable inference or probabilistic loss components.

• Interface:  
  – The primary method is evaluate_answer(answer, expected, question_type) that returns a boolean decision (True for a correct answer, False otherwise).  
  – The decision is derived through two separate evaluation strategies:
   a. Deterministic evaluation for label-specific questions  
   b. ChatGPT-aided evaluation for general questions

──────────────────────────────
2. Revised Evaluation for Label-Specific Questions (evaluate_label_specific)

Previous Strategy Issue:  
 – Sole reliance on strict regex (e.g., using “\byes\b”) may fail to capture valid affirmative answers with extra contextual phrases (e.g., “I suppose yes” or “definitely, yes”).

Revised Strategy:  
 • Preprocess the answer using robust text cleaning (convert to lowercase, remove extraneous punctuation, and normalize whitespace).  
 • Instead of a single strict regex match for “yes,” adopt a two-step strategy:  
  1. Use a regex that captures common affirmative tokens along with optional surrounding context. For example, a pattern that checks for a standalone “yes” but also accepts variants like “i suppose yes”, “yeah, yes”, “sure, yes” by using an approach that allows additional words before or after the keyword if they do not conflict with a negative indicator.  
  2. Simultaneously search for any occurrence of negative terms using a regex (e.g., “\bno\b” or similar).  
 • Define a set of affirmative keywords (configurable via eval_config loaded from config.yaml) that list acceptable words such as:
   – {"yes", "yeah", "yep", "indeed", "correct", "affirmative"}  
 • Decide “correct” if at least one affirmative keyword is found in an unambiguous context and no strong negative indicator is present.  
 • For ambiguous phrasing (e.g., “I suppose yes”), the matching strategy should flag these cases as uncertain and—depending on a configurable tolerance setting—either accept them as positive or classify them as ambiguous for further analysis. The exact behavior (for instance, a fuzzy match threshold) is set via configuration.

3. Revised Evaluation for General Questions (evaluate_general)

Previous Strategy Issue:  
 – Direct use of the ChatGPT API without a well-defined fallback for ambiguous responses may lead to instability. There is a lack of clear retry limits, timeout handling, and resolution for responses containing both “yes” and “no” or neither.

Revised Strategy:  
 • Construct a clear evaluation prompt that instructs ChatGPT to return strictly “yes” or “no.” Include explicit wording such as:  
  "Please answer only with 'yes' or 'no'."  
 • When sending the prompt, use configuration parameters (loaded from config.yaml under, e.g., an 'api' or 'eval' section) that define:  
  – The maximum number of retries (e.g., retry_limit: 3)  
  – A timeout for the API call  
  – A default fallback value (e.g., default_result: False)  
 • After receiving the API response, apply robust text cleaning and use regex (with word-boundaries) to extract binary keywords.  
 • If the response is ambiguous (for instance, if both “yes” and “no” are detected, or if neither clear term appears), trigger a retry (up to the maximum retry limit).  
 • If after retries the response remains ambiguous, log a warning and return the default fallback value specified in configuration.  
 • This strategy ensures that any instability or non-binary output is handled in a predefined manner to maintain reproducibility.

4. Configuration Integration (evaluate_answer and Overall)

Previous Strategy Issue:  
 – The previous analysis did not clearly specify how parameters from config.yaml are integrated into each evaluation function.

Revised Integration:  
 • During initialization of the VAE class, pass an evaluation configuration dictionary (eval_config) that is loaded, validated, and injected by the main configuration loader (e.g., from config.yaml). This configuration should include:  
  – Tolerance settings for fuzzy matching in label-specific evaluation (e.g., acceptable affirmative patterns, negative patterns, and thresholds)  
  – API settings for ChatGPT evaluation such as model name ("gpt-3.5-turbo"), retry limits, timeout durations, and fallback defaults  
 • Within the methods evaluate_label_specific and evaluate_general, explicitly reference these configuration parameters to drive decisions (i.e., do not hard-code patterns or retry values). For example:  
  – Affirmative_patterns = eval_config.get("affirmative_patterns", [r"\byes\b", ...])  
  – retry_limit = eval_config.get("api_retry_limit", 3)  
 • In evaluate_answer, select the evaluation method based on the question type and pass the necessary configuration values along to the helper functions.

5. Detailed Pseudocode Flow for VAE Methods

A. evaluate_label_specific(answer: str) -> bool  
 1. Clean the answer text.  
 2. Iterate over a list of configurable affirmative regex patterns (e.g., patterns including word boundaries but allowing extra context).  
 3. If at least one affirmative pattern matches and no configurable strong negative pattern (such as “\bno\b”) is matched in a conflicting context, return True.  
 4. Otherwise, return False.  
 5. Optionally, if ambiguous matching is detected, use a configured threshold to decide (or log the ambiguity).

B. evaluate_general(answer: str, expected: str) -> bool  
 1. Construct the prompt including clear instructions (“Answer only with 'yes' or 'no'”) and include answer and expected details.  
 2. Initialize retry count from configuration.  
 3. While retry count > 0:  
  a. Send prompt to ChatGPT API with timeout handling as specified by config; catch exceptions and handle timeout.  
  b. Clean the response text and use regex to detect a clear “yes” or “no”.  
  c. If the response is unambiguous (only one of the two is present), return the corresponding boolean.  
  d. Otherwise, decrement retry count and optionally adjust the prompt or log the ambiguous response.  
 4. If retries are exhausted, return the configured default fallback value and log a warning.

C. evaluate_answer(answer: str, expected: str, question_type: str) -> bool  
 1. Load any required configuration parameters (passed to the VAE instance on initialization).  
 2. If question_type equals “label_specific”, call evaluate_label_specific(answer) and return the result.  
 3. Else if question_type equals “general”, call evaluate_general(answer, expected) and return the output.  
 4. For unknown question types, log an error and return the default fallback value.
