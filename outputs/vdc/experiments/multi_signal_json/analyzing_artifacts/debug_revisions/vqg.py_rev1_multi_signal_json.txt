Below is a detailed logic analysis for the vqg.py module. This module implements the Visual Question Generation (VQG) class, which is responsible for producing the questions used later in the VDC detection pipeline. The VQG class must generate two types of questions for each given sample based on its label:

──────────────────────────────
1. OVERVIEW

• The VQG module is designed to generate:
  – General Questions: Fixed, template-based questions (e.g., “Describe the image briefly.”). These are independent of the image label and are drawn from a predefined list.
  – Label-Specific Questions: Questions tailored to the given label. These are generated by calling the ChatGPT API (using the GPT‑3.5‑turbo model per our config) with a prompt that instructs the LLM to produce insightful, label-specific questions.

• Both types of questions are later aggregated (vote-based) along with the answers from the VQA module, and then the VAE module uses these responses to compute a matching score for detecting dirty samples.

──────────────────────────────
2. INITIALIZATION (__init__)

• The VQG class should be initialized with:
  – A list of general question templates. This can be hard-coded or loaded from a configuration.
  – A dictionary (or similar structure) of label-specific prompt templates.
    ▪ The configuration file (config.yaml) specifies the required number of label-specific questions for each dataset:
      • CIFAR-10: 4 questions
      • ImageNet-100: 6 questions
      • ImageNet-Dog: 4 questions
    ▪ It is useful to store a specific prompt (or set of prompts) that will instruct ChatGPT to generate these label-specific questions.
  – (Optional) Any additional configuration settings if needed. For example, you might include the current dataset name to look up the correct count.
  – A cache (for example, a dictionary) to store previously generated label-specific questions keyed by label. This caching helps to avoid redundant API calls and handle rate limits.

──────────────────────────────
3. METHOD: generate_general_questions(label: str) -> List[str]

• Although the input parameter “label” is provided for interface consistency, the general questions are not dependent on the label.
• Steps:
  1. Retrieve the predefined list of general question templates.
  2. Use the “general_questions_count” from the configuration (in config.yaml it is set to 2) and select that many questions from the list.  
     ▪ You may use random sampling from the full list or always choose the first N templates.
  3. Return the selected list.

• Edge Considerations:
  – If the predefined list has fewer questions than required, then return what is available.
  – Ensure any randomness (if used) has a reproducible seed mechanism (potentially using a utility function from utils.py).

──────────────────────────────
4. METHOD: generate_label_specific_questions(label: str) -> List[str]

• Purpose: Generate a set of questions tailored to the provided label.
• Steps:
  1. Determine the number of label-specific questions to generate.
     ▪ This count is retrieved from the config under detection → label_specific_questions.
     ▪ Either the caller (or the VQG instance) knows the active dataset (e.g., CIFAR-10, ImageNet-100, or ImageNet-Dog) so that it can select the correct question count.
  2. Check if there is already a cache entry for this label.
     ▪ If yes, return the cached questions.
  3. If not cached:
     a. Construct a prompt by combining the label with a predefined prompt template. The prompt should instruct ChatGPT to “generate X label‑specific visual questions to identify if the label of the image is correct.” For instance, for label "airplane", the prompt might include sample questions like “Is the object in the image designed for flight?” or similar.
     b. Use a utility function (from utils.py) to call the ChatGPT API. This function should handle API request details, rate limiting, and caching of responses.
     c. Parse the ChatGPT API response to extract a list of questions.
         – The output might be a plain text block with line breaks or bullet points. Use simple string splitting or regular expressions to extract each question.
     d. Validate that the number of questions meets the required count. If less than expected, consider either repeating some or using fallback prompts.
     e. Store the extracted list in the cache keyed by the label.
  4. Return the list of label-specific questions.

• Error Handling:
  – If the ChatGPT API call fails or the returned output is empty/unexpected, log the error (using logging from utils.py) and return an empty list or a fallback list based on predefined templates.
    
──────────────────────────────
5. DEPENDENCIES & UTILITIES

• The VQG module will depend on:
  – The openai package (as specified in config) for ChatGPT API calls.
  – Utility functions from utils.py for:
     ▪ Caching API responses.
     ▪ Performing string cleaning and parsing of API output.
     ▪ (Possibly) handling random sampling with set seeds.
• The module must load the configuration values (e.g., general_questions_count and label_specific_questions per dataset) – these values come directly from config.yaml.

──────────────────────────────
6. SUMMARY OF FUNCTIONAL FLOW

For each sample (using its label):
  a. Call generate_general_questions(label) → returns a list of predetermined general questions.
  b. Call generate_label_specific_questions(label) → returns a list of questions produced by ChatGPT tailored to the label.
  c. Combine both lists if needed for further processing in the detection pipeline.
  
This process ensures that every sample is associated with a set of questions which will then be passed on to the VQA module for answering and subsequently evaluated in the VAE module using vote-based ensemble.

──────────────────────────────
7. CONSIDERATIONS

• Reproducibility: Ensure that random selections (if any) use a fixed seed so that the experimental results can be reproduced.
• API Rate Limits and Caching: Since label-specific questions are obtained via ChatGPT, effective caching is essential to avoid unnecessary API calls.
• Robust Parsing: The module should gracefully handle variations in the API response format.

This logical structure and step-by-step breakdown in vqg.py will ensure that the Visual Question Generation module faithfully adheres to the VDC design and supports experiments as described in the paper, using the configuration from config.yaml and interfacing properly with other modules via consistent APIs.
