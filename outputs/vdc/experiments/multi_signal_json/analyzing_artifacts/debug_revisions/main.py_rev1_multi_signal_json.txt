# Logic Analysis for main.py

1. Reading Configuration and Initialization
   • Begin by loading the configuration from the “config.yaml” file. This will bring in key settings such as:
       – Training hyperparameters (epochs, learning_rate, batch_size values for each dataset, learning rate decay schedule).
       – Detection settings (threshold value, number of general and label‐specific questions per dataset).
       – Dirty sample generation parameters (noisy ratio, poisoning samples_per_class for CIFAR-10, ImageNet-100, and ImageNet-Dog).
       – Model identifiers (which detector and classifier are used).
       – API related configurations (ChatGPT model “gpt-3.5-turbo” and Instruct-BLIP model).
   • Initialize any random seed settings (possibly via a utility function) so that results are reproducible.

2. Dataset Loading and Dirty Sample Injection
   • Instantiate the DatasetLoader class with the loaded configuration. This module will:
       – Load the image dataset(s) (CIFAR-10, ImageNet-100, or ImageNet-Dog) using torchvision.
       – Preprocess images (resizing/normalization as needed) in a consistent manner.
   • Call the loader’s load_data() method to obtain a dictionary containing the training and test splits.
   • Use the inject_dirty_samples() function from the DatasetLoader (or its linked utility functions) to simulate dirty samples:
       – For poisoning: Apply backdoor triggers according to the parameters in dirty_sample_generation (e.g., add BadNets or Blended triggers on CIFAR-10 as configured).
       – For noisy labels: Randomly flip labels (symmetric or asymmetric noise) with the specified noisy ratio (η = 0.4).
       – Optionally, include hybrid dirty samples (if both poison and noise are injected).
   • Log the number of samples injected per class and overall dirty sample ratio for future reference.

3. Module Instantiations for the Detection Pipeline
   • Instantiate the Visual Question Generation (VQG) module:
       – Provide it with a list of pre-defined general question templates.
       – Pass in the label-specific question prompt templates (these should be cached or pre-obtained via ChatGPT API; configuration tells how many questions to generate for the current dataset).
   • Instantiate the Visual Question Answering (VQA) module:
       – Initialize it with the Instruct-BLIP model configuration (ensuring correct input pre-processing for images).
   • Instantiate the Visual Answer Evaluation (VAE) module:
       – Provide any evaluation configuration (e.g., expected answer evaluation strategy – using string matching for label-specific and ChatGPT for general questions).
   • Instantiate the VDCDetector class by passing the VQG, VQA, and VAE instances along with the detection threshold (set to 0.5 from config).
       – The detector, for each sample, will generate questions, obtain answers, evaluate responses, and calculate a matching score (vote-based ensemble).

4. Detection over the Training Dataset (Purification Process)
   • Loop over each sample (image and its noisy/poisoned label) in the training dataset:
       – Call vdc_detector.detect_sample(image, label) for each sample.
           > Inside this method, the following should occur:
              ▪ VQG generates both general questions (fixed set per sample, count = 2) and label-specific questions (number depends on dataset, e.g., 4 for CIFAR-10, 6 for ImageNet-100, 4 for ImageNet-Dog).
              ▪ For each generated question, the VQA module is used to produce an answer by processing the input image.
              ▪ The VAE module then evaluates each answer against an expected answer strategy. For label-specific questions, a simple check (e.g., presence of “yes” or “no”) is performed; for general questions, a prompt is sent to ChatGPT to decide if the answer aligns with the expected semantic content.
              ▪ The votes (booleans) are aggregated to compute a matching score, s = (votes / total_questions).
       – If s < 0.5 (as specified by the threshold), mark the sample as “dirty.”
   • After processing all samples, filter out any sample flagged as dirty. This purification yields a revised training dataset that is expected to have a lower ratio of dirty samples.
   • Optionally, log detection metrics (like the number of samples flagged and if available the ground truth for TPR and FPR from the simulated injection process).

5. Training the Classifier on the Purified Dataset
   • Instantiate the Trainer class, passing:
       – The purified training dataset.
       – The classifier model which is “ResNet-18” (the model identifier is provided in config).
       – Hyperparameters: epochs = 100, optimizer “SGD,” learning rate = 0.1, batch size based on dataset (e.g., 128 for CIFAR-10) and the learning rate decay schedule (e.g., reduce after 50 and 75 epochs for CIFAR-10).
   • Call the train() method on Trainer:
       – This method will perform a standard training loop over the purified dataset.
       – The training loop includes forward passes, loss computation, backpropagation using SGD, and manual adjustments of learning rates according to the decay schedule.
       – Periodically log intermediate metrics (such as training loss, accuracy).
   • At the end of training, obtain the final trained ResNet-18 model.

6. Model Evaluation
   • Instantiate the Evaluation class with:
       – The trained classifier model.
       – The test dataset (which should be clean by design).
   • Call evaluate() to compute:
       – Clean Accuracy (ACC): The percentage of correct predictions on clean test samples.
       – Attack Success Rate (ASR): For datasets that originally had backdoor triggers, evaluate whether the triggers still cause misclassification.
       – (If available) Additional detection metrics might include TPR and FPR on the purified dataset.
   • Log and report the computed evaluation metrics.

7. Final Experiment Orchestration and Logging
   • At the end of the experiment flow, main.py should collect all the metrics (training loss, ACC, ASR, detection performance) and print or log them in a structured format.
   • Ensure caching strategies are in place for API calls (both ChatGPT and Instruct-BLIP) to reduce redundant calls and maintain reproducibility.
   • Also, ensure that error handling and proper cleanup occur for API connections.
   • Finally, allow main.py to be re-run easily (for different dataset choices such as CIFAR-10, ImageNet-100, or ImageNet-Dog) by either passing command-line parameters or modifying the configuration file.

Summary of the Main Flow in main.py:

• Load configuration parameters from config.yaml.
• Create DatasetLoader instance → load datasets → inject dirty samples.
• Instantiate VQG (with fixed templates and dynamic label prompts), VQA (with Instruct-BLIP), and VAE (using ChatGPT evaluation and string matching) modules.
• Create a VDCDetector using these modules and the threshold (0.5).
• For each training sample, run detection; remove samples that do not meet the matching score threshold.
• With the purified (clean) training set, use Trainer to retrain a ResNet-18 classifier using defined hyperparameters.
• Finally, evaluate the trained model on the test set via Evaluation to compute ACC, ASR, and (if applicable) detection metrics.
• Log and present final results.

This analysis ensures that the code in main.py faithfully reproduces the experimental design laid out in the paper, follows the modular structure defined by our design, and consistently uses the configuration settings from config.yaml.
