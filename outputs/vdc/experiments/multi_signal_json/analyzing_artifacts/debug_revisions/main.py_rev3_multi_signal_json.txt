# Revised Logic Analysis for main.py

1. Configuration Loading and Initialization  
   • Read all configuration parameters from “config.yaml.” These include:  
       – Training hyperparameters: epochs, learning_rate, batch_size (per dataset), and learning rate decay schedule.  
       – Detection settings: the detection threshold (which may differ from 0.5 if specified in config), the number of general questions (e.g., 2) and label-specific questions (e.g., 4 for CIFAR-10, 6 for ImageNet-100, 4 for ImageNet-Dog).  
       – Dirty sample generation parameters: noisy_ratio (0.4) and poisoning samples per class for each dataset.  
       – API configuration for ChatGPT (model: gpt-3.5-turbo) and Instruct-BLIP for VQA.  
   • Initialize a random seed (via a shared utility) to ensure reproducibility.  
   • Set up any caching or API rate-limit management systems for repeated external calls.

2. Dataset Loading and Dirty Sample Injection  
   • Instantiate the DatasetLoader class with the configuration. This module is responsible for loading datasets (CIFAR-10, ImageNet-100, ImageNet-Dog) via torchvision and performing image preprocessing (resizing, normalization, etc.).  
   • Call load_data() to obtain the training and testing split.  
   • Call inject_dirty_samples() to embed dirty samples into the training data. Here, special attention is given to hybrid dirty sample creation:
       – For poisoning, a backdoor trigger (e.g., a 3×3 white square for CIFAR-10, other appropriate triggers for ImageNet subsets) is applied based on the configuration (e.g., 50 or 500 samples per class).  
       – For noisy labels, a specified proportion (η = 0.4) of the labels are flipped using either a symmetric or an asymmetric strategy.  
       – If hybrid dirty samples are desired, the injection process follows a fixed order. For example, define a mode flag (e.g., hybrid_mode) that, when enabled, first applies poisoning triggers to a designated subset of samples and then applies noisy label perturbation on either the same samples or on a separate disjoint subset.  
         ▪ The order is explicitly defined: either (a) first inject the trigger (which forces a target label) and then, if the sample is marked for additional noise, the noisy label injection is bypassed (to avoid conflict) or (b) apply the noisy labels first and then override with the poisoning trigger if designated.  
         ▪ This explicit procedure and conflict-resolution mechanism must be documented so that any change in configuration results in a consistent definition.  
       – Log clear statistics (e.g., number of samples poisoned only, noised only, and hybrid cases) to aid in reproducibility.

3. Module Instantiation for the Detection Pipeline  
   • Instantiate the Visual Question Generation (VQG) module with:  
       – A fixed list of predefined general question templates (e.g., “Describe the image briefly.”).  
       – Label-specific question prompts retrieved or generated by ChatGPT, with the number per dataset read from configuration.  
   • Instantiate the Visual Question Answering (VQA) module using the Instruct-BLIP model; ensure that image preprocessing adheres strictly to its input requirements.  
   • Instantiate the Visual Answer Evaluation (VAE) module, whose responsibilities are:
       – For deterministic (label-specific) questions, performing string matching (e.g., checking for “yes” after lowercasing and trimming punctuation).  
       – For general questions, sending a specially designed evaluation prompt to ChatGPT to output a “yes” or “no” decision.
       – Normalizing both sources of evaluation into a binary vote (1 for yes, 0 for no).
   • Create the VDCDetector module by passing the VQG, VQA, and VAE instances along with the detection threshold that is read from the configuration (instead of hardcoding 0.5).  
       – In detect_sample, for each generated question the resulting binary vote is collected.  
       – The matching score is computed as s = (weighted sum of votes) / (total weight). By default, all votes have equal weight (typically weight 1), but this formulation allows future calibration adjustments without changing the overall architectural design.

4. Detection and Purification (Training) Phase  
   • Loop over every sample in the training dataset and call VDCDetector.detect_sample(image, label):
       – VQG produces both general and label-specific questions based on the current label.  
       – VQA answers each question using the Instruct-BLIP model.  
       – VAE evaluates these answers (binary yes/no values using the appropriate evaluation strategy).  
       – The overall matching score s is computed from these binary votes.  
       – A sample is flagged as dirty if s < (configured threshold), ensuring that the threshold can be adjusted from the configuration.
   • After processing all samples, filter the training data by removing the samples flagged as dirty.  
   • Log the detection statistics (e.g., number of detected dirty samples, and if ground truth is available, detection TPR/FPR).

5. Training the Classifier  
   • Instantiate the Trainer class using the purified dataset and instantiate the ResNet-18 classifier model.  
       – Pass in hyperparameters from the configuration (e.g., 100 epochs, SGD with learning_rate: 0.1, and appropriate batch sizes from config such as 128 for CIFAR-10 or 64 for ImageNet sets).  
       – Ensure that the learning rate decay is scheduled according to the provided epochs (e.g., 50 and 75 for CIFAR-10, or 30 and 60 for ImageNet datasets).
   • Execute the Trainer.train() loop, where training metrics (loss, accuracy) are logged periodically.

6. Evaluation (ASR and ACC Computation)  
   • Instantiate the Evaluation module with the trained classifier and the clean test dataset.  
   • For standard accuracy (ACC), evaluate the classifier on the original clean test set.  
   • For Attack Success Rate (ASR), generate a triggered test subset by:  
       – Applying the same trigger injection procedure used during training on a fixed number of test samples.  
       – Use identical trigger parameters (e.g., same pattern, same position, and same blending/overlay parameters) to ensure consistency.  
       – Specify in the analysis that a predetermined number or percent (e.g., 10% per class or the same number as configured for poisoning in training) of test samples are modified.  
   • Compute the ASR as the percentage of triggered test samples that are misclassified to the predefined target class.  
   • Log and return both ACC (from clean samples) and ASR (from triggered samples) to allow a clear comparison with the experimental results in the paper.

7. Final Orchestration and Logging  
   • After training and evaluation, collect final metrics (ACC, ASR, and if available detection metrics such as TPR and FPR over training samples).  
   • Print or save the metrics in a clear, structured format.  
   • Ensure any API connections are properly closed (or cached responses are properly saved) and that cleanup routines are executed.

Summary of the Revised Main Flow:

• Configuration is loaded from config.yaml (with detection threshold, number of questions, noisy and poisoning parameters).  
• DatasetLoader loads and preprocesses the dataset, then injects dirty samples with a clearly defined order for hybrid dirty sample creation: either by sequentially applying poisoning triggers and then noisy label injection (or vice versa) with explicit conflict resolution.  
• VQG generates both general and label-specific questions, VQA produces responses using Instruct-BLIP, and VAE evaluates each response to produce binary votes – all votes are then equally weighted (with scope for future calibration) and used to compute a matching score s. The decision to flag a sample relies on a threshold read from the configuration.  
• The training set is purified by removing samples with s below the threshold.  
• The Trainer then retrains ResNet-18 on the purified set with hyperparameters defined in config.yaml.  
• The Evaluation module computes ACC on the clean test set and ASR on a triggered test subset generated by applying the same backdoor trigger parameters as in training. The number and selection of test samples for ASR are explicitly specified for consistency.  
• Final results are logged and printed, ensuring proper reproducibility according to the paper’s design.

This revised analysis explicitly addresses the ambiguities regarding hybrid sample creation (by stipulating the order and conflict resolution), clarifies that the detection threshold is taken from configuration (with equal weighting of votes by default), and details the procedure for generating the triggered test subset for consistent ASR evaluation.
