Below is the revised logic analysis for vqg.py that addresses the critique regarding the parsing of ChatGPT API responses in the generate_label_specific_questions method.

──────────────────────────────
1. OVERVIEW

• The VQG module is responsible for producing both:
  – General Questions: These are fixed, template-based questions (e.g., “Describe the image briefly.”) selected from a predefined list.
  – Label-Specific Questions: These are tailored to the given label and are generated by invoking the ChatGPT API. The prompt instructs the LLM to return several insightful questions customized to the label.

• The generated questions are later used in the VDC detection pipeline (passed to the VQA module and evaluated by VAE).

──────────────────────────────
2. INITIALIZATION (__init__)

• On initialization, the VQG class should set up:
  – A list of predefined general question templates.
  – A configuration for label-specific question prompts (e.g., a base prompt that is combined dynamically with the label).
  – The desired number of label-specific questions to generate per dataset (this value is provided by the config.yaml file; for example, CIFAR-10: 4, ImageNet-100: 6, ImageNet-Dog: 4).
  – An internal cache (e.g., a dictionary) mapping labels to their previously generated questions, which helps to minimize API calls and cope with rate limits.

──────────────────────────────
3. METHOD: generate_general_questions(label: str) -> List[str]

• Functionality:
  – Ignores the given label (for interface consistency) and simply returns a fixed set of general questions.
  – Uses the “general_questions_count” from config.yaml (set to 2) to select the required number of questions.
  – May select questions from the list in a fixed order or via a reproducible random mechanism (with a fixed seed).

──────────────────────────────
4. METHOD: generate_label_specific_questions(label: str) -> List[str]

• Purpose:
  – Generate a set of label-specific questions for a given label by querying the ChatGPT API.

• Revised parsing approach (to address the critique):
  1. Determine the number of questions needed from the configuration based on the active dataset.
  2. Check the cache for the given label; if questions are already available, return them.
  3. If not available:
     a. Construct a prompt for ChatGPT that explicitly instructs it to generate the desired number of label-specific questions. For example, the prompt can specify:
        • “For the label ‘{label}’, please generate N insightful visual questions that, when answered, help in verifying whether an image truly belongs to this label. Format your output as a JSON array of strings.”
        
        This directive to output valid JSON can help enforce a structured response.
        
     b. Call the ChatGPT API via a utility function (from utils.py) that handles network calls, caching of responses, and potential rate limits.
     c. Parse the API response using a robust multi-stage approach:
        – First, attempt to parse the response as JSON. If the response is valid JSON (i.e., a list of strings), use this list directly.
        – If JSON parsing fails (for example, if the response is plain text or not properly structured), then fall back on a robust heuristic parsing:
          • Normalize the text (trim spaces, handle different line endings).
          • Search for bullet points by looking for common markers (e.g., “- ”, “* ”, or enumerated items like “1. ”).
          • Use a regular expression that identifies sequences ending with a question mark “?” (ensuring the extraction of complete questions) rather than a simple string split.
          • Optionally, if multiple candidate strings are identified, filter out any that do not meet a minimal length criterion or do not end in “?”.
     d. Validate the parsed list:
        – Check if the number of questions meets (or exceeds) the required count.
        – If fewer questions are extracted than expected, log a warning and use additional fallback strategies (such as reissuing the API call with a modified prompt or appending default template questions).
     e. Store the final list in the cache (keyed by the label) for future calls.
  4. Return the parsed and validated list of label-specific questions.

• Key Points:
  – The revised strategy explicitly encourages output as a JSON array via the prompt. This minimizes reliance on brittle string splitting.
  – A multi-stage parsing process is employed:
       • Primary: JSON parsing.
       • Secondary: Regular expressions and heuristic extraction of question patterns.
  – Robust error checking and fallbacks are integrated to handle variability in the API response format.

──────────────────────────────
5. DEPENDENCIES & UTILITIES

• The VQG module depends on:
  – The openai package (for ChatGPT API calls) with caching functions from utils.py.
  – Utility functions for robust response parsing and string cleaning (also likely provided by utils.py).
• It loads configuration settings (number of questions, prompt templates) from config.yaml.

──────────────────────────────
6. FUNCTIONAL FLOW & USAGE

For a given label:
  a. generate_general_questions(label) is invoked to retrieve a fixed set of template-driven questions.
  b. generate_label_specific_questions(label) is invoked to obtain a set of tailored questions.
     – This function ensures that responses from ChatGPT are parsed using a structured and robust approach (first trying JSON parsing, then heuristic extraction).
  c. The combined list (general plus label-specific) is returned to the higher-level detection pipeline.

──────────────────────────────
7. FINAL REMARKS

• The revised plan for generate_label_specific_questions improves robustness against variations in ChatGPT responses by:
  – Requesting a structured JSON answer.
  – Implementing multi-layered parsing with JSON-first and heuristic fallbacks.
  – Incorporating validation and fallback strategies to ensure the required number of questions is produced.
  
This refined logic ensures that the VQG module can reliably generate and extract label-specific questions, aligning with the overall VDC design and addressing the critique regarding fragile parsing methods.
