Below is the further revised logic analysis for detection.py that now explicitly addresses the handling of expected answers for label‐specific questions.

──────────────────────────────
1. OVERALL CLASS FUNCTIONALITY

The VDCDetector class’s role is to integrate the three modules:
 • VQG (Visual Question Generation)
 • VQA (Visual Question Answering)
 • VAE (Visual Answer Evaluation)

It determines whether an input sample (image, label) is “dirty” by:
 a. Generating a set of questions (both general and label‐specific),
 b. Obtaining corresponding answers from a multimodal model,
 c. Evaluating each answer using a vote-based approach,
 d. Computing a matching score and comparing it against a threshold (0.5 as specified).

──────────────────────────────
2. CLASS INITIALIZATION

• The __init__ method accepts instances of the VQG, VQA, and VAE modules along with the threshold (from config.yaml).
• These are stored as internal members (e.g., self.vqg, self.vqa, self.vae, self.threshold) for later use during detection.
• No additional state is maintained beyond what is provided by the configuration.

──────────────────────────────
3. DETAILED DETECTION PROCESS (detect_sample Method)

A. Question Generation via VQG

 1. Two Distinct Sets of Questions Are Generated:
  a. General Questions:
   – Call VQG.generate_general_questions(label) to obtain a list of template-based questions (for example, “Describe the image briefly.”).
   – These questions capture a holistic description and later require prompt-based evaluation.
  b. Label-Specific Questions:
   – Call VQG.generate_label_specific_questions(label) to generate questions tailored to the provided label.
   – Crucially, for each label-specific question, the expected answer is determined beforehand. There are two potential strategies:
    • Embed the expected answer directly with the question (e.g., as a tuple or dictionary such as { "question": ..., "expected_answer": "yes", "type": "label-specific" }).
    • Retrieve the expected answer from a configuration or a pre-generated cache that maps each question to its deterministic expected answer.
   – This explicit linking ensures that when the VAE evaluates label-specific questions, it knows precisely what answer it should look for when performing string matching.
 2. The two sets are then merged into a single list while preserving their “question_type” tags (e.g., "general" or "label-specific") together with the expected answer for label-specific items.

B. Answer Generation via VQA

 1. Iterate over each question item.
 2. For every question, invoke VQA.answer_question(image, question_text) to retrieve an answer.
 3. The answer, along with its associated question type and expected answer (if applicable), is recorded for evaluation.

C. Answer Evaluation via VAE

 1. For Each Question–Answer Pair:
  a. If the question’s type is "label-specific":
   – The answer is evaluated using a deterministic method.
   – VAE.evaluate_answer receives not only the actual answer and the question type but also the expected_answer (which was either embedded with the question or retrieved from configuration).
   – The evaluation method then performs simple string matching (after any necessary text normalization, such as converting to lowercase) to determine if the answer meets the expected criteria (e.g., containing “yes”).
  b. If the question’s type is "general":
   – Since no fixed expected answer exists, VAE.evaluate_answer builds an evaluation prompt and calls ChatGPT via its API.
   – The returned binary decision (yes/no) indicates whether the answer aligns with the semantics expected for a “clean” sample.
 2. Each evaluated answer yields a Boolean value (True if correct; False otherwise).

D. Matching Score Calculation and Decision

 1. Sum the Boolean “votes” (True answers) for all questions.
 2. Compute the matching score sᵢ = (number of correct evaluations) / (total number of questions).
 3. Compare the matching score with the threshold (0.5):
  – If sᵢ < 0.5, mark the sample as “dirty” (indicating visual-linguistic inconsistency).
  – Otherwise, mark it as “clean.”
 4. Return the final detection decision along with additional details (such as the matching score and per-question evaluations) for debugging or further analysis.

 – Log generated questions (with their types and expected answers for label-specific items), obtained answers, and evaluation results.

• Modularity:
 – Ensure that each module (VQG, VQA, VAE) exposes only the defined public interfaces.
 – All configurable parameters (such as number of questions, threshold, and expected answer mappings) are sourced from config.yaml or associated configuration logic.

──────────────────────────────
5. SUMMARY OF THE REVISED FLOW

For each sample (image, label):
 a. The VQG module generates two sets of questions:
  – General questions (requires prompt-based ChatGPT evaluation).
  – Label-specific questions that include an associated, predefined expected answer (embedded with the question or retrieved).
 b. VQA provides an answer for each question.
 c. VAE evaluates each answer:
  – For general questions by querying ChatGPT.
  – For label-specific questions via deterministic string matching against the supplied expected answer.
 d. The matching score is computed (votes/total questions).
 e. If the matching score is below the threshold (0.5), the sample is flagged as “dirty.”
 f. The decision and evaluation details are returned for further processing.
