Below is the revised logic analysis for the “utils.py” module. This updated analysis addresses the previous critiques by clarifying ambiguous variables and parameters, and by linking certain trigger-related parameters to the configuration file. The revised analysis for each main section is as follows:

──────────────────────────────
1. Configuration Loading and Seed Initialization  
 • load_config(config_path):  
  – Reads the YAML configuration file (config.yaml) and returns a dictionary with all settings (training, detection, dirty sample generation, models, and API details).  
 • init_seed(seed):  
  – Sets the random seed for reproducibility. In addition to Python’s random, NumPy, and Torch seeds, it now explicitly seeds GPU experiments by calling torch.cuda.manual_seed_all(seed).  
  – Example calls include random.seed(seed), numpy.random.seed(seed), torch.manual_seed(seed), and torch.cuda.manual_seed_all(seed).

──────────────────────────────
2. Image Preprocessing Utilities  
 • preprocess_image(image, dataset_name):  
  – Converts input images into the format expected by downstream modules.  
  – Applies resizing, normalization, and tensor conversion using torchvision transforms.  
  – The exact parameters (e.g., image dimensions, normalization mean and std) are chosen based on the dataset name (CIFAR-10, ImageNet-100, ImageNet-Dog), either hard-coded or loaded from the configuration.  
 • tensor_to_image(tensor):  
  – Optionally converts a Torch tensor back to a PIL image for visualization or debugging.

──────────────────────────────
3. Trigger Overlay Functions for Backdoor Attacks  
These functions inject various triggers into an image. Each function now explicitly defines or links critical parameters to avoid ambiguity:

 • overlay_badnets_trigger(image, dataset_name):  
  – Injects a visible backdoor trigger. For CIFAR‑10, it places a 3×3 white square at the lower‐right corner; for ImageNet datasets, it applies a similar patch (for example, a 21×21 white square) scaled to the image dimensions.  
 • overlay_blended_trigger(image, trigger_image, blend_ratio=0.1):  
  – Blends a given “Hello Kitty” trigger image with the input image using the provided blend ratio.  
 • apply_sig_trigger(image, delta=20, frequency=6):  
  – Implements the SIG attack by overlaying a horizontal sinusoidal signal.  
  – The formula used is v(i, j) = Δ sin(2π·j·f/m), where the variable m is now explicitly defined as the number of columns (i.e., the image width) after preprocessing. This clarification ensures consistency with the paper.  
 • apply_trojanNN_trigger(image, trigger_mask):  
  – Overlays a predefined trigger mask (e.g., an Apple logo) on the image.  
 • apply_ssba_trigger(image, ssba_params):  
  – Implements a SSBA-style trigger.  
  – Instead of using an undefined parameter, this function now accepts a clearly named parameter (ssba_params), a dictionary that should be populated from configuration (or defined with default values) covering details such as the U-Net model settings, decoder configuration, iteration count, etc.  
  – A note is added that if SSBA is computationally intensive, a simplified simulation might be used at first.  
 • apply_wanet_trigger(image, warping_params):  
  – Applies a warping transformation (elastic image warping) to simulate the WaNet attack.  
  – The parameter warping_params is now described as a defined dictionary that includes parameters like the warping intensity and scale. These parameters should ideally be provided via the configuration file or set to default values, ensuring the transformation is consistent with the experimental design described in the paper.

4. API Caching, Logging, and Text Utilities  
Since external API calls (ChatGPT for question generation and evaluation, Instruct-BLIP for VQA) are involved:
 • cache_api_response(key, response):  
  – Stores an API response under a unique key (for instance, a hash of the prompt and parameters) to help reduce duplicate requests.  
 • get_cached_response(key):  
  – Retrieves a cached API response if available.  
 • generate_api_cache_key(input_params):  
  – Generates a stable, unique key from the input parameters by serializing and hashing them; ensures repeated prompts hit the cache.  
 • setup_logging(log_file=None):  
  – Configures the Python logging module with timestamps and log levels; logs can be sent to a file or the console.  
 • clean_text(text):  
  – Standardizes text by lowercasing, removing punctuation, and trimming extra whitespace—this function is essential prior to evaluating answers with simple string matching.  
 • simple_similarity(text1, text2):  
  – Optionally computes a basic similarity measure between two strings which can help when comparing responses in the visual answer evaluation module.
