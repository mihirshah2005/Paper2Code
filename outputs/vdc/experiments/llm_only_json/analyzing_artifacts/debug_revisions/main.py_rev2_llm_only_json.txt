Revised Logic Analysis for main.py

1. Configuration Loading and Initialization  
   • Load experimental settings from “config.yaml” via a YAML parser.  
     – Under “training”: retrieve epochs, optimizer ("SGD"), learning_rate (0.1), batch_size (dataset-specific), and the learning rate decay schedule.  
     – Under “detection”: retrieve threshold (0.5), general_questions_count (2), and number of label-specific questions for each dataset.  
     – Under “dirty_sample_generation”: retrieve noisy_ratio (0.4) and the poisoning samples per class for each dataset.  
     – Under “api”: load model names (ChatGPT “gpt-3.5-turbo” and Instruct-BLIP).  
   • Set a global random seed early (via a helper function in utils.py) to be used in data loading, dirty sample injection, model initialization, and training. This ensures reproducibility across the entire pipeline.

2. Dataset Loading and Dirty Sample Injection  
   • Instantiate the DatasetLoader with the configuration settings.  
   • Call DatasetLoader.load_data() to load the training and test splits for the chosen dataset (e.g., CIFAR-10, ImageNet-100, or ImageNet-Dog).  
     – Ensure that images are preprocessed (resized, normalized) as required by subsequent models.  
   • Call DatasetLoader.inject_dirty_samples(data) to inject dirty samples.  
     – Ensure that during this injection process, a ground truth flag (for example, “is_dirty” or a corresponding label field) is maintained with each sample. This flag records whether a sample was originally clean or intentionally modified (either via backdoor poisoning, noisy labels, or a hybrid combination) so that later evaluation of detection metrics (TPR, FPR) can compare the detector’s decisions against this ground truth.  
     – For poisoning, follow the settings from the configuration (e.g., for CIFAR-10: 50 and 500 samples per class).  
     – For noisy labels, apply the specified noisy ratio (0.4) by flipping labels accordingly.  
     – In the case of hybrid dirty samples, clearly define and document a strategy that, for example, (a) first injects poisoning triggers on a subset of samples and then (b) further modifies a different fraction by flipping their labels (or vice versa). This design should be consistent so that both modifications can be tracked via the sample’s ground truth status.

3. Detector Module Initialization  
   • Initialize the Visual Question Generation (VQG) module:  
     – Pass the pre-defined general question templates and label-specific prompt templates.  
     – Use the number of label-specific questions as specified in config.detection.label_specific_questions for the current dataset.  
   • Initialize the Visual Question Answering (VQA) module:  
     – Use the settings from “api.instruct_blip” (i.e., select Instruct-BLIP) to answer generated questions.  
     – Ensure that proper image preprocessing is applied per the Instruct-BLIP requirements.  
   • Initialize the Visual Answer Evaluation (VAE) module:  
     – For deterministic label-specific questions, implement simple string-based matching (checking for the presence of “yes” or “no” after lowercasing and cleaning the text).  
     – For general questions, design the evaluation process with the following improvements:  
         • Instead of making repeated real-time ChatGPT API calls (which can produce non-deterministic outcomes and be subject to rate limiting), incorporate caching strategies to store API responses based on input prompts and answers.  
         • Optionally, consider a fallback local evaluation method (e.g., using a similarity metric or a pre-trained local language model) to ensure scalability and reproducibility.  
   • With VQG, VQA, and VAE ready, instantiate the VDCDetector module by passing these components along with the detection threshold (0.5 from config.detection.threshold).

4. Dirty Sample Detection and Dataset Purification  
   • For each sample (image and its label) in the training dataset, run the following steps:  
     – Call VDCDetector.detect_sample(image, label) which internally:  
       ▪ Uses VQG to generate the union of two types of questions (general and label-specific), noting that expected answers should be derived from deterministic string matching or a predefined set when available.  
       ▪ For each question, it calls VQA.answer_question(image, question) to retrieve an answer.  
       ▪ For each answer, call VAE.evaluate_answer(answer, expected_answer, question_type); ensure that for general questions the evaluation output is cached from ChatGPT/API calls.  
       ▪ Compute the matching score as the ratio of correct evaluations (votes) to the total number of questions.  
       ▪ Return a clean/dirty decision based on whether the matching score meets or exceeds α = 0.5.  
   • When processing, log every sample’s ground truth “dirty” flag (set during injection) alongside the detector’s decision. This log is essential for the later computation of TPR (true positive rate) and FPR (false positive rate).  
   • Filter out all samples detected as dirty to create the purified training dataset.

5. Classifier Training on the Purified Dataset  
   • Instantiate the Trainer module by providing:  
     – The chosen classifier (ResNet-18, as specified in config.models.classifier).  
     – The purified training dataset (which has been cleaned based on detection decisions).  
     – The hyperparameters from the configuration: epochs (100), batch size (dataset-specific), learning rate (0.1), and the learning rate decay schedule.  
   • The Trainer sets up the training loop (with SGD) ensuring that seed initialization is used to control randomness (by calling the global seed setting function from utils.py if not already set).  
   • Call Trainer.train() to retrain the ResNet-18 classifier using the purified dataset.

6. Model Evaluation  
   • Instantiate the Evaluation module with the following components:  
     – The trained classifier from the Trainer module.  
     – The test dataset loaded earlier.  
     – A clear definition for the test dataset: if evaluating ASR (attack success rate) on poisoned samples, then the test dataset should either be pre-augmented with known backdoor triggers (using the same injection mechanism) or include a held-out set of intentionally poisoned samples to simulate the attack scenario.  
   • Call Evaluation.evaluate() to compute the evaluation metrics:  
     – For detection: Compare detector decisions (from the log maintained during injection) with the ground truth flags to compute TPR and FPR.  
     – For retraining: Calculate ASR on the test set for samples with backdoor triggers, and compute overall Clean Accuracy (ACC) on clean samples.  
   • Log or print these metrics for analysis.

7. Orchestration and Final Remarks  
   • The main.py script orchestrates the overall experiment by invoking:  
     – Configuration loading (ensuring random seed initialization, so that every module uses the same seed).  
     – Data loading along with dirty sample injection (with ground truth maintenance for later metric computation).  
     – Detector initialization and invocation for sample-wise detection (coupled with caching for ChatGPT-dependent evaluations).  
     – Retraining of the classifier using the purified dataset, with hyperparameter scheduling as per config.yaml.  
     – Final evaluation of both the detector and the purified classifier (ensuring that the test dataset includes simulated poisoned samples for ASR evaluation).  
   • In addition, address potential API rate limiting by:  
     – Implementing caching within VQG and VAE modules for all ChatGPT API responses.  
     – Optionally, exploring local alternative evaluation strategies if API calls prove too time‐consuming or inconsistent.  
   • Finally, log all experimental outcomes (including the number of dirty samples injected vs. detected, TPR, FPR, ASR, ACC) to support reproducibility and further tuning.
