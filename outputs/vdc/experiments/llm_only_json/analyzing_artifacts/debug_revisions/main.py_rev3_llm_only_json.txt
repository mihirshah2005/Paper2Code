Revised Logic Analysis for main.py

1. Configuration Loading and Global Initialization  
   • Read settings from config.yaml using a YAML parser. In particular, load:
     – Training hyperparameters (epochs, optimizer, learning_rate, batch_size per dataset, and lr_decay_schedule).
     – Detection parameters (threshold = 0.5, general_questions_count = 2, and number of label-specific questions per dataset, e.g., 4 for CIFAR-10 and ImageNet-Dog, 6 for ImageNet-100).
     – Dirty sample generation parameters (noisy_ratio = 0.4 and poisoning samples per class for each dataset).  
   • It is crucial to set a global random seed (via a helper function in utils.py) early. This seed must be applied consistently across data-loading, dirty sample injection, trigger generation, model initialization, and training to ensure full reproducibility.

2. Dataset Loading and Dirty Sample Injection  
   • Instantiate the DatasetLoader with the configuration settings.
   • Call DatasetLoader.load_data() to load the training and testing splits (for CIFAR-10, ImageNet-100, or ImageNet-Dog).  
     – Apply any required image preprocessing (resizing, normalization) to support downstream modules.
   • Invoke DatasetLoader.inject_dirty_samples(data) to simulate dirty samples:
     – For pure poisoning, inject backdoor triggers using methods defined in utils.py (e.g., BadNets, Blended, SIG, etc.).  
         ▪ For example, for CIFAR-10 use a 3×3 white square at the lower-right corner for BadNets, following the paper’s design.
     – For noisy labels, flip labels based on the noisy_ratio (0.4) using either symmetric or asymmetric strategies.
     – For hybrid dirty samples, define a clear sequential strategy (or mixture ratio) as follows:
         ▪ First (optionally) apply the poisoning trigger injection to a subset of samples.
         ▪ Then apply noisy label modifications either on the same samples (if simulating a hybrid attack) or on a different fraction, as defined in the configuration.
         ▪ In both cases, mark or flag each sample with a ground truth indicator (for example, an “is_dirty” field that distinguishes clean, poisoned, noisy-label, or hybrid samples). This ground truth is later used to compute detection metrics such as TPR and FPR.
     – Ensure that the configuration explicitly controls the order and mixture ratios to avoid ambiguity and to align with the paper’s experimental setup.

3. Detector Module Initialization  
   • Instantiate the Visual Question Generation (VQG) module:
     – Provide it with a fixed set of general question templates (e.g., “Describe the image briefly”, “Describe the image in detail”) and label-specific prompt templates for generating tailored questions via ChatGPT.  
     – Note: Expected answers for label-specific questions should be pre-defined or generated offline and stored so that the subsequent evaluation is deterministic.
   • Instantiate the Visual Question Answering (VQA) module:
     – Set it up with the multimodal model specified in config.api.instruct_blip (i.e., Instruct-BLIP).  
     – Ensure that images are preprocessed in the format required by the model.
   • Instantiate the Visual Answer Evaluation (VAE) module:
     – For deterministic label-specific questions, use an enhanced string matching approach. Rather than a raw “yes/no” match, consider applying text cleaning and even a fuzzy-matching algorithm to account for phrasing nuances.
     – For general questions, design the evaluation to:
         ▪ Construct an evaluation prompt and call ChatGPT.
         ▪ Use a caching mechanism: define caching keys (for instance, based on hashing the tuple (question, answer, expected answer) along with a unique identifier) and store API responses in a local dictionary or file-based cache.
         ▪ Specify a fallback local evaluation method (e.g., using cosine similarity of embeddings if the API call fails or if the cached response is absent) to ensure reproducibility and reduce API rate limiting.
     • With VQG, VQA, and VAE set up, instantiate the VDCDetector by passing these modules along with the detection threshold (0.5, from config.detection.threshold).  
     • In VDCDetector.detect_sample, the matching score is computed as:
         ▪ For each generated question, obtain a boolean judgment (True/False) indicating if the answer is “correct” based on VAE.evaluate_answer.
         ▪ The expected answers should be pre-defined (for label-specific questions) or obtained from a stored reference.
         ▪ Calculate the matching score as (number of True evaluations) / (total number of questions).
         ▪ Note: Consider adopting fuzzy matching techniques for string comparisons to handle variations in phrasing.

4. Dirty Sample Detection and Data Purification  
   • For every sample in the training dataset, call VDCDetector.detect_sample(image, label). Internally:
     – VQG generates both general and label-specific questions.
     – VQA provides answers for each question, and VAE evaluates them—using the caching mechanism for ChatGPT API calls to improve consistency.
     – The detector then computes a matching score.
     – Compare the matching score to the threshold (0.5) and mark the sample as “dirty” if the score is below this value.
   • Maintain a log that maps each sample’s unique identifier with its ground truth “dirty” flag (from the injection stage) and the detector’s decision. This log is essential for calculating TPR and FPR later.
   • Purify the training dataset by removing samples flagged as dirty based on the detector's decision.

5. Classifier Training on the Purified Dataset  
   • Instantiate the Trainer module. Fetch the classifier model from config.models.classifier. (If the configuration does not match “ResNet-18”, log a warning or adjust accordingly so that the intended model is used.)
   • Provide the purified training dataset along with hyperparameters from the configuration (epochs: 100, batch_size: as per dataset, learning_rate: 0.1, and the specified lr_decay_schedule).
   • Execute Trainer.train() to retrain the classifier on the purified dataset, ensuring that the random seed is applied to all stochastic components.

6. Model Evaluation  
   • Instantiate the Evaluation module with the trained classifier and the test dataset.
   • To compute ASR (Attack Success Rate), ensure that the test dataset is concretely structured as follows:
         – Generate or pre-augment a test set that includes poisoned samples by applying the same backdoor trigger injection method used during training.
         – Maintain consistency between the poisoning mechanism in the training injection and the test augmentation; this may involve using a held-out set that is modified using the same function in utils.py.
   • Evaluation.evaluate() computes:
         – Detection metrics (TPR and FPR) by comparing detected dirty samples against the ground truth labels maintained during injection.
         – Retraining metrics on the test set, including ASR on the poisoned samples (i.e., the fraction of samples where the attack is successful) and clean accuracy (ACC) on clean samples.
   • Log and output these metrics for comprehensive analysis.

7. Orchestration and Final Remarks  
   • The main function orchestrates the overall pipeline by sequentially:
         – Loading configuration and initializing the random seed.
         – Loading the dataset and injecting dirty samples while maintaining ground truth markers.
         – Initializing and configuring the VQG, VQA, and VAE modules and integrating these into the VDCDetector.
         – Running detection on all training samples and purifying the dataset accordingly.
         – Training the classifier with the purified dataset as specified and ensuring the model choice aligns with config.models.classifier.
         – Evaluating the model and detector on a carefully structured test set that includes simulated poisoned samples, providing clear ASR and ACC metrics.
   • In addition, the design accounts for:
         – A detailed caching strategy in VAE.evaluate_answer to ensure that every API call to ChatGPT is cached with keys based on (question, answer, expected answer, dataset identifier) and a fallback mechanism (e.g., local similarity measure) is in place if an API call is not possible.
         – A clearly defined hybrid injection strategy that specifies whether the poisoning and noisy label modifications are applied sequentially (e.g., first alter the image for poisoning, then adjust the label) or independently. The chosen strategy should be clearly codified in the configuration and documented within the injection function.
         – Detailed trigger design for poisoning: Refer to utils.py for concrete functions that implement the backdoor triggers (e.g., the BadNets design uses a 3×3 white square, Blended uses a “Hello Kitty” image with a blend ratio of 0.1, SIG uses a sinusoidal signal with ∆ = 20 and f = 6, etc.). These details ensure consistency with the original paper.
