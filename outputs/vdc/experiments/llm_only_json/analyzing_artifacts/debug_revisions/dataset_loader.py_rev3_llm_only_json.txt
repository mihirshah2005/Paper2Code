Below is the revised logic analysis for dataset_loader.py. The analysis now clarifies several design decisions and explicitly addresses all the critiques concerning reproducibility, configuration-driven behavior, exclusive modification ordering, edge-case handling, augmentation selection, and distributed loading. Each section below outlines the revised logic for the corresponding functions in the DatasetLoader class.

1. Initialization: DatasetLoader.__init__

• Configuration and Seed Control  
  - The __init__ method will explicitly look for a configuration key (e.g., "seed") in the configuration dictionary passed to the class. If the key is not provided, a default seed (for example, 42) will be used.  
  - This random seed will be used for all subsequent random operations within this module (such as class selection, sample selection for dirty injections, etc.).  
  - The initialization will also store the dataset type (e.g., "CIFAR-10", "ImageNet-100", or "ImageNet-Dog") and any corresponding dataset-specific configuration (e.g., a key for custom class subsets if provided by the user).  
  - The __init__ method sets up two image transformation pipelines:
     ▪ A standard training/testing pipeline that includes common data augmentations (random cropping, flipping, normalization).  
     ▪ A “trigger-preserving” pipeline that is applied to samples that receive poisoning (backdoor triggers). This pipeline is designed with either fewer spatial transformations or uses a metadata flag (e.g., adding an extra attribute on the sample) to ensure that aggressive augmentations do not modify or remove the injected trigger.
  - Note: In distributed or multi-worker scenarios, the __init__ method should also ensure that a base seed is configured. Additionally, when using torch DataLoader with multiple workers, a worker_init_fn (or similar mechanism) will be defined (elsewhere in the codebase) to offset the seed per worker. This ensures that all workers share a consistent random seed context, preserving reproducibility.

2. Loading Data via load_data

• Deterministic and Configurable Selection  
  - The load_data() function uses torchvision to load datasets:
     ▪ For CIFAR-10, load directly using torchvision.datasets.CIFAR10 with the designated training and testing transforms.
     ▪ For ImageNet-100 and ImageNet-Dog:
         – The method first checks the configuration for an optional key (for example, "subset_classes") that specifies the exact classes to use.  
         – If no such key is provided, the loader deterministically selects classes using the fixed random seed:  
             • For ImageNet-100, it selects 100 classes randomly (using a sorted and fixed order after random selection) to ensure reproducibility.  
             • For ImageNet-Dog, if custom dog classes are not provided, a pre‐defined list of 10 dog classes is used.
         – Within each selected class, a fixed number of samples are chosen for training and testing using a sorted order or a deterministic random sampling (again, seeded appropriately). For example, training split: 500 images per class for ImageNet-100 (or 800 for ImageNet-Dog) and test split: 100 per class (or 200, respectively).
  - The function logs the list of selected classes and indices (or filenames) to allow verification and reproducibility.
  - Finally, load_data() constructs and returns a dictionary (or similar structure) with keys "train_data" and "test_data", where each sample is represented consistently as (image, label).

3. Injecting Dirty Samples via inject_dirty_samples

• Clear Ordering and Exclusivity Between Modifications  
  - The inject_dirty_samples(data) method first partitions the training dataset per-class into disjoint subsets to ensure that poisoning (trigger injection) and noisy label modifications are applied exclusively (unless a hybrid mode is explicitly desired via configuration).  
     ▪ Option 1 (Mutually Exclusive):  
         – For each class, select samples to be poisoned first based on the configured number of poison samples (read from the configuration’s poisoning sub-key for the dataset).  
         – Remove these poisoned samples from the pool before selecting samples for noisy label injection.  
         – Apply noisy label injection to the remaining samples up to the specified noisy_ratio.  
     ▪ Option 2 (Hybrid, if desired):  
         – A fixed operation order is defined (for example, always apply trigger injection first, then apply noisy label modification on samples that have not been poisoned).  
         – In our design we recommend the mutually exclusive approach for clarity and consistency.
         
• Edge-Case Handling When Sample Counts Are Insufficient  
  - The method will check, for each class, if there are sufficient samples to meet the combined quotas for poisoning and noisy label modification.  
  - If the number of available samples is less than what is requested by the configuration, the method will:
         – Apply modifications to as many samples as are available and log a detailed warning regarding the quota not being met.
         – This ensures that the injection process never raises a runtime error due to insufficient samples and that experimental reproducibility is maintained.
  
• Detailed Mechanism for Modifications  
  - For poisoning:
         – For each chosen sample for poisoning, call a specific utility function (e.g., inject_badnets, inject_blended, inject_SIG, etc.) defined in utils.py with a clearly defined interface.  
         – These utility functions will accept the image and required parameters (such as trigger size, position, blend ratio) and return the modified image.  
         – Once a trigger is injected, update the sample’s label to the target label (e.g., class 0).
  - For noisy label injection:
         – The method distinguishes between symmetric and asymmetric noise.  
         – For symmetric noise, a sample’s label is randomly (but deterministically via the fixed seed) changed to one of the other labels.  
         – For asymmetric noise, the new label is set using a clearly defined function: for example, new_label = (original_label + 1) mod K. Note that if K is the total number of classes, then the function is implemented precisely to follow the paper’s intended behavior.
  - Document and log the number of samples modified and the type of modification applied for later validation.

4. Data Augmentation Pipeline and Trigger Preservation

• Mechanism for Switching Augmentation Pipelines  
  - The DatasetLoader will include a strategy to mark samples that have been modified with a trigger injection.  
  - For example, each sample data structure may include an extra metadata flag (e.g., "trigger_injected": True) so that downstream processing (or the DataLoader’s collate_fn) can switch or apply a “trigger-preserving” augmentation pipeline for that sample.  
  - This mechanism prevents standard aggressive augmentations (like random cropping or flipping) from distorting the injected trigger.

5. Handling Randomness in Distributed Data Loading

• Ensuring Consistent Seeding Across Workers  
  - In multi-worker scenarios (where torch.utils.data.DataLoader may spawn several processes), the DatasetLoader's random seed should be shared by using worker_init_fn in the DataLoader.  
  - This worker_init_fn should seed each worker with a combination of the main seed and the worker’s ID so that all worker processes produce deterministic and reproducible behavior.

6. Summary of Revised Workflow

• __init__:  
  - Reads the "seed" from the configuration (or uses the default) and sets torch and numpy random seeds.  
  - Reads any dataset-specific class subsets from the configuration (if provided); otherwise, defaults to deterministic selection.  
  - Initializes both standard and trigger-preserving transformations, potentially marking samples with a metadata flag for pipeline selection.

• load_data():  
  - Loads the dataset using torchvision.  
  - For ImageNet-100 and ImageNet-Dog, deterministically selects classes based on either configuration-specified subsets or default random (but seeded) selections, and then deterministically selects training/testing samples per class.  
  - Returns a dictionary with “train_data” and “test_data,” with detailed logging for reproducibility.

• inject_dirty_samples():  
  - Partitions each class of the training set into disjoint groups to ensure no sample undergoes both poisoning and noisy label injection (unless hybrid mode is explicitly selected).  
  - For poisoning: Uses configuration values to determine the number of samples; calls well-defined utility functions (with explicitly documented interfaces) to inject backdoor triggers; and updates labels to the target label.  
  - For noisy label injection: Applies modifications to the remaining samples, using a clearly defined mapping for asymmetric noise and random selection for symmetric noise, all under a fixed seed.  
  - Handles edge cases by checking if the available samples in a class meet the quota and logs warnings if modifications are applied to fewer samples than requested.

• Distributed Loading Consideration:  
  - The design includes measures (e.g., worker_init_fn) to ensure that random seed propagation is maintained across multiple worker processes for reproducibility.

Conclusion

This revised analysis explicitly links the random seed behavior to a configuration key, clarifies how class subsets for ImageNet-based datasets are selected (and how the user can provide custom subsets), and defines a strict ordering for modifications between poisoning and noisy label injection (with disjoint partitions and clear edge-case handling). It further outlines a mechanism (using metadata flags) to select a trigger-preserving data augmentation pipeline and notes considerations for distributed data loading. These revisions ensure alignment with the original paper’s reproducibility requirements and eliminate ambiguity in experimental setups.
