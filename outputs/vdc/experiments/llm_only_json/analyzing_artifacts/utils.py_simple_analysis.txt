Below is the revised, detailed logic analysis for utils.py. The analysis is organized by the core responsibilities of the module:

──────────────────────────────
1. Configuration Loading and Seed Initialization  
 • load_config(config_path)
  – Reads the YAML configuration file (config.yaml) and returns a Python dictionary containing all settings (training, detection, dirty_sample_generation, models, api).  
  – Ensures that any parameter (e.g., number of label-specific questions, patch sizes if needed) is centralized.  
 • init_seed(seed)
  – Sets the random seed to guarantee reproducibility across the entire system.
  – Calls:
   ○ random.seed(seed)
   ○ numpy.random.seed(seed)
   ○ torch.manual_seed(seed)
   ○ torch.cuda.manual_seed_all(seed)
  – In addition, to ensure full determinism for GPU operations, it sets:
   ○ torch.backends.cudnn.deterministic = True
   ○ torch.backends.cudnn.benchmark = False
  – This configuration avoids non-deterministic behavior in CUDA and fully aligns with reproducible experiment requirements.

──────────────────────────────
2. Image Preprocessing Utilities  
 • preprocess_image(image, dataset_name)
  – Converts input images to the proper format (e.g., PIL to tensor conversion) and applies resizing, normalization, and any other required preprocessing.
  – Chooses normalization parameters (mean, std) and target dimensions based on dataset_name (for example, CIFAR-10 vs ImageNet variants); these values can be either hard-coded defaults or read from the configuration.
 • tensor_to_image(tensor)
  – Optionally converts a Torch tensor back into a PIL image for visualization or debugging.

3. Trigger Overlay Functions for Backdoor Attacks  
Each trigger injection function is now designed to avoid hard-coded or ambiguous parameters by either (a) computing values dynamically based on the image dimensions, or (b) retrieving critical parameters directly from the configuration file. For each function:

 • overlay_badnets_trigger(image, dataset_name)
  – Injects a visible backdoor trigger.
  – For CIFAR-10, the standard patch is a 3×3 white square placed in the lower-right corner. For ImageNet datasets, the standard patch has dimensions 21×21.
  – To handle images of non-standard dimensions, the function now optionally computes a patch size as a fraction of the image width and height (e.g., 10% of the width if the image size deviates from the standard) while defaulting to standard values defined in the configuration.
  – The position of the patch (e.g., lower-right) is also parameterizable through the configuration to ensure consistency with the experimental design.

 • overlay_blended_trigger(image, trigger_image, blend_ratio=0.1)
  – Blends a provided “Hello Kitty” trigger image with the original image using a blend_ratio.
  – The blend_ratio is a parameter that can also be read from the configuration, ensuring consistency across experiments.

 • apply_sig_trigger(image, delta=20, frequency=6)
  – Implements the SIG attack by overlaying a horizontal sinusoidal signal on the image.
  – The formula used is: v(i, j) = Δ sin(2π · j · f / m), where Δ is delta, f is the frequency, and m is explicitly defined as the number of columns (i.e., the image width after preprocessing).
  – This clear definition of ‘m’ avoids ambiguity and ensures that the signal is computed consistently regardless of image resolution.

 • apply_trojanNN_trigger(image, trigger_mask)
  – Overlays a predefined trigger mask (e.g., an Apple logo) onto the image.
  – The trigger_mask should now be specified as an external file or object loaded from the configuration file (e.g., a file path provided in config.yaml). This ties the definition of the trigger_mask to a reproducible source and avoids undefined behavior.
  – The function may include steps to preprocess the trigger_mask to match the image dimensions if necessary.

 • apply_ssba_trigger(image, ssba_params)
  – Implements the SSBA-style trigger injection.
  – Instead of a vague additional_params argument, this function clearly accepts a dictionary (ssba_params) whose structure is strictly defined. For example, ssba_params should include keys such as:
   ○ "u_net_settings": parameters for the U-Net model (architecture details, pretrained weights if any),
   ○ "decoder_config": settings for the spatial transformer or decoder network,
   ○ "iteration_count": the number of iterations to run,
   ○ (Additional keys as needed)
  – These ssba_params are expected to be either defined directly in the configuration file (under a dedicated key like dirty_sample_generation.ssba_params) or passed as a fully specified dictionary. This explicit structure ensures reproducibility and a clear mapping to the experimental design described in the paper.

 • apply_wanet_trigger(image, warping_params)
  – Applies an elastic warping transformation to simulate the WaNet attack.
  – The warping_params parameter is now expected to be a defined dictionary, with keys such as:
   ○ "intensity": a value representing the strength of the warping,
   ○ "scale": a factor to determine how much the image is distorted,
   ○ (Other possible keys as required)
  – These settings are intended to be supplied via the configuration file (e.g., under dirty_sample_generation.warping_params), ensuring the transformation is consistent with the intended experimental design.

4. API Caching, Logging, and Text Utilities  
To support external API calls (ChatGPT and Instruct-BLIP) and robust logging:
 • cache_api_response(key, response)
  – Saves an API response mapped to a unique key (which can be generated by hashing the request parameters) to prevent redundant API calls.
 • get_cached_response(key)
  – Retrieves a cached response if one exists, else returns None.
 • generate_api_cache_key(input_params)
  – Generates a unique key for a given set of API input parameters by serializing and hashing them. This ensures consistency and reproducibility across API calls.
 • setup_logging(log_file=None)
  – Sets up a logging configuration (using Python’s logging module) that logs timestamps, log levels, and messages to either a file or the console.
 • clean_text(text)
  – Processes text by lowercasing, stripping punctuation, and removing extra spaces. This utility is essential for deterministic string matching in answer evaluations.
 • simple_similarity(text1, text2)
  – Optionally computes a simple similarity score between two texts, which can be used to further support the visual answer evaluation in ambiguous cases.
