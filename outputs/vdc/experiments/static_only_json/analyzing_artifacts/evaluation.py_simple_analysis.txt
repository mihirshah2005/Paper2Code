# evaluation.py – Logic Analysis

Overview:
-----------
The Evaluation class will be initialized with two main inputs:
 • model – the already trained classifier (typically a ResNet‑18 instance) from trainer.py.
 • test_data – a dictionary or similar data structure (loaded by dataset_loader.py) that contains:
      ▪ Clean test images and corresponding ground‐truth labels.
      ▪ (Optionally) A set of test images modified with backdoor triggers (to evaluate ASR).
      ▪ (Optionally) Ground truth binary labels for “dirty” status to compute detection metrics (TPR/FPR) if the detection phase is evaluated separately.

The evaluate() method in this class is tasked with the following:
 1. Putting the model into evaluation mode and iterating over the test data.
 2. For each test sample, use the model to compute predictions.
 3. Compute different metrics depending on the test subset:
    • Clean Accuracy (ACC):
         – For the subset of “clean” test samples, compare the predicted class to the true label.
         – ACC is calculated as the number of correct predictions divided by the total number of clean test samples.
    • Attack Success Rate (ASR):
         – For the subset of test samples that have been modified with the attack trigger, the “correct” outcome is defined as the model outputting the target label (the backdoor target).
         – ASR is given by: (# of triggered samples predicted as the target label) divided by (total number of triggered samples).
    • Detection Metrics – True Positive Rate (TPR) and False Positive Rate (FPR):
         – If the test data carries detection-level labels from the VDCDetector (i.e., each sample is annotated with its ground truth “dirty” status), then:
               • TPR = (Number of dirty samples correctly detected as dirty) / (Total number of actual dirty samples)
               • FPR = (Number of clean samples incorrectly marked as dirty) / (Total number of clean samples)
         – This part is applicable when the evaluation is intended to gauge the quality of the VDC cleaning step. It relies on the detector’s output flags that might have been saved alongside the test set.
 4. Collate all these computed metrics into a dictionary and return it so that downstream logging or reporting can be performed.

Important Implementation Details:
-------------------------------------
• Data Iteration & Preprocessing:
   - Before inference, the model should be set to eval() mode (and gradients disabled with torch.no_grad()).
   - Use the same image preprocessing pipeline as was used during training (e.g., resizing, normalization), as provided by the DatasetLoader and utils.py.
 
• Metrics for Classification (ACC and ASR):
   - Clean ACC: Iterate over clean test samples (each sample is a pair: (image, true label)). For each sample, forward-pass through the classifier; compare the predicted label (e.g., using argmax on output logits) with the ground-truth label.
   - ASR: For triggered/poisoned test samples, the target label is predetermined by the backdoor attack setup (e.g., “class 0” as mentioned in the paper). Compute the proportion of triggered samples that have been classified as the target.
   - These metrics are directly reported as percentages.
 
• Metrics for Detection (TPR, FPR):
   - If the test_data contains extra flags indicating whether a sample is “dirty” (e.g., injected with a backdoor or noisy label) versus “clean”, then compute:
         • True Positives (TP): dirty samples correctly identified (by VDC) as dirty.
         • False Negatives (FN): dirty samples misclassified as clean.
         • False Positives (FP): clean samples wrongly flagged as dirty.
         • True Negatives (TN): clean samples correctly left unflagged.
   - From these counts, compute:
         • TPR = TP / (TP + FN)
         • FPR = FP / (FP + TN)
 
• Handling Backdoor/Triggered Test Samples:
   - The evaluation may require generating triggered versions of test images if not already provided in test_data.
   - A utility function (from utils.py) can be called for each clean image to simulate a backdoor trigger for ASR evaluation. This function should apply the same trigger transformation used during dirty sample generation.
 
• Integration with Configurations:
   - Although the Evaluation module does not directly modify training hyperparameters, it should be aware of the experimental setup. For example, knowing that the classifier is trained on a purified dataset, it should expect high ACC on clean samples and low ASR on triggered samples if purification worked correctly.
   - The detection threshold (α = 0.5) and other parameters (e.g., poison/noise ratios) are defined in config.yaml but are not directly used in the evaluation class—they are more relevant during the detection and training phases.
 
• Design Considerations for Reproducibility:
   - Set random seeds where needed to maintain consistency.
   - Batch the test data according to the same batch size used during training (from config.yaml) for efficiency.
   - Ensure that the output dictionary contains a standardized set of keys (e.g., "ACC_clean", "ASR_poisoned", "TPR_detection", "FPR_detection") so that downstream analysis or plotting routines can easily log and compare results.
 
• Return Value:
   - The evaluate() method should finally return a dictionary of metrics. For example:
         {
           "ACC_clean": 93.5,  # percentage accuracy on unpoisoned test data
           "ASR_poisoned": 1.2,  # percentage of triggered test samples classified as the target
           "TPR_detection": 99.5,  # percentage true positive rate (if detection evaluation is performed)
           "FPR_detection": 2.8,   # percentage false positive rate (if detection evaluation is performed)
         }

Summary of the Evaluation Flow (Sequence Diagram Perspective):
---------------------------------------------------------------
1. Instantiate the Evaluation class with the trained model and test dataset.
2. Set the model to evaluation mode.
3. For each test sample in the “clean” subset:
     • Process the image and predict label.
     • Compare with ground truth and count correct predictions to obtain ACC.
4. For each test sample in the “triggered/poisoned” subset:
     • Process the (triggered) image and predict label.
     • Check if the predicted label equals the predetermined attack target.
     • Count successful attack cases to compute the ASR.
5. (Optional) If test samples include ground truth dirty/clean flags from detection:
     • Compare the detector’s decision with the ground truth.
     • Compute TPR (for dirty samples) and FPR (for clean samples).
6. Return a summary dictionary containing all metrics.

Final Remarks:
---------------
• The Evaluation class serves as the final step in the experimental pipeline.
• It leverages the model (ResNet‑18) that was retrained on the purified dataset after the VDC cleaning stage.
• Consistent image preprocess and careful handling of both clean and triggered samples are crucial for correct metric calculations.
• Depending on the experimental scenario (pure classification vs. detection evaluation), some metrics (TPR/FPR) might be absent or computed differently.
• Any caching of predictions, batching strategy, and random seed configuration must be consistent to enable reproducibility.
