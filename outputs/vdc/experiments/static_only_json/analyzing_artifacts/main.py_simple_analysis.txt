Below is the detailed logic analysis for main.py. This analysis outlines the step‐by‐step orchestration that the main entry point must perform to reproduce the experiments as described in the paper and the provided plan. It explains the flow—from configuration loading and dataset preparation to dirty sample detection, purification, retraining, and evaluation—while explicitly referring to the values set in config.yaml.

──────────────────────────────
1. Load Configuration and Initialize Modules

• Read the configuration file (config.yaml) to extract all necessary settings:
  – Training settings: epochs (100), optimizer ("SGD"), learning rate (0.1), batch sizes (128 for CIFAR-10, 64 for ImageNet-100 and ImageNet-Dog), and learning rate decay schedules (epoch 50 & 75 for CIFAR-10; epoch 30 & 60 for ImageNet-100 and ImageNet-Dog).
  – Detection settings: detection threshold (0.5), general_questions_count (2), and label_specific_questions count for each dataset (4 for CIFAR-10 and ImageNet-Dog, 6 for ImageNet-100).
  – Dirty sample generation: noisy_ratio (0.4) and poisoning samples per class (e.g., 50 or 500 for CIFAR-10; 5 or 50 for ImageNet-100; 80 for ImageNet-Dog).
  – Model names: “VDC” as the detector and “ResNet-18” for the classifier.
  – API settings for ChatGPT (gpt-3.5-turbo) and Instruct-BLIP.

• Instantiate utility functions (from utils.py) to load and parse the config (also setting up random seeds for reproducibility).

──────────────────────────────
2. Load and Prepare the Dataset

• Instantiate the DatasetLoader (from dataset_loader.py) by passing the configuration.
  – Load the three datasets (CIFAR-10, ImageNet-100, and ImageNet-Dog) using TorchVision.
  – Perform necessary image preprocessing and splitting into train and test sets.
  
• Inject dirty samples into the training set:
  – For poisoned samples: Based on the config poisoning parameters, use attack functions (e.g., BadNets, Blended, SIG, etc.) to modify images and alter labels.
  – For noisy labels: Flip labels in a symmetric (uniform random flip to any other label) or asymmetric manner (flip to (i mod K)+1) with a ratio of 0.4.
  – The exact functions for trigger injections are implemented in utils.py and referenced by DatasetLoader.
  
• The DatasetLoader returns a dictionary including:
  – train_data: with both clean and artificially injected dirty samples.
  – test_data: as the standard test split.

──────────────────────────────
3. Initialize the VDC Detection Pipeline

• Instantiate the VQG module:
  – Using predefined general question templates (e.g., “Describe the image briefly.”) and also ChatGPT-based label-specific prompts.
  – The number of label-specific questions is read from config (4 for CIFAR-10/ ImageNet-Dog, 6 for ImageNet-100).

• Instantiate the VQA module:
  – Wrap the Instruct-BLIP model (as specified under models.api.instruct_blip) to process images and answer questions.
  – Ensure that the necessary image preprocessing (e.g., resizing, normalization) is applied.

• Instantiate the VAE module:
  – For evaluating answers from the VQA:
      ▪ For label-specific questions: perform string matching (e.g., check for “yes” keyword, after text normalization).
      ▪ For general questions: call ChatGPT (gpt-3.5-turbo) with an evaluation prompt to judge semantic consistency.
  – The VAE returns a boolean result (True if the answer is “correct” and False otherwise).

• Instantiate the VDCDetector by passing the VQG, VQA, VAE modules, and the detection threshold (0.5 from config).

──────────────────────────────
4. Perform Dirty Sample Detection and Purification

• Loop over each sample (image, label) in the train_data:
  – Call the VDCDetector.detect_sample(image, label).
    ▪ Within this method:
         • Call VQG.generate_general_questions(label) and VQG.generate_label_specific_questions(label)
         • For each generated question (from both sets), pass the image and question to VQA.answer_question(image, question)
         • Evaluate each answer using VAE.evaluate_answer(answer, expected_answer, question_type)
         • Count the “votes” (i.e., correctly evaluated answers) and compute the matching score:  
             sᵢ = (# of correct answers) / (total number of questions)
         • Compare sᵢ with the threshold α (0.5). If sᵢ < 0.5, flag the sample as dirty.
  – Record the decision for each sample (clean vs. dirty), and then remove dirty samples from the dataset.
  
• The purified dataset (clean samples only) is then passed along for training.

──────────────────────────────
5. Retrain the Classifier on the Purified Dataset

• Instantiate the Trainer (from trainer.py) with:
  – The purified training dataset obtained after detection.
  – The ResNet-18 classifier.
  – Hyperparameters from config (epochs=100, learning rate=0.1, optimizer=SGD, and the dataset-specific batch size and LR decay schedule).
  
• Run the training loop in Trainer.train():
  – Use SGD optimizer, monitor training loss, accuracy, and adjust learning rate at the specified epochs.
  – Log the training progress.
  
• Optionally, also train/retrain under different dirty sample conditions (e.g., varying poisoning ratio or noisy ratio) as per the experiments.

──────────────────────────────
6. Evaluate the Trained Classifier

• Instantiate the Evaluation module (from evaluation.py) by providing:
  – The trained ResNet-18 model.
  – The test_data from DatasetLoader (which is clean).
  
• Call Evaluation.evaluate(), which will:
  – Compute the Clean Accuracy (ACC) on the test set.
  – If applicable, compute the Attack Success Rate (ASR) for the backdoor attack scenarios.
  – Additionally, if needed, compute detection metrics: TPR (how many dirty samples have been correctly detected) and FPR (how many clean samples were flagged mistakenly).

• Print and log the evaluation metrics. These results will be compared with the baselines as reported in the paper.

──────────────────────────────
7. Final Orchestration and Logging

• In main.py, after each major step (dataset loading, detection/purification, training, evaluation), log the outcomes to the console and to any log files.
• Ensure that all randomness is controlled (e.g., using fixed seeds via utils) so that the experiments are reproducible.
• Handle API caching (for ChatGPT and Instruct-BLIP calls) to minimize API call variance and to respect rate limits.
• Wrap the entire flow in a main() function and call it when executing main.py.

Conclusion

The main.py entry point serves as the experiment orchestrator. It:
  • Initializes configurations and module instances (DatasetLoader, VDCDetector, Trainer, Evaluation),
  • Loads and “dirties” the training set,
  • Runs the VDC pipeline to remove dirty samples (using a vote‐based ensemble over answers from both general and label‐specific questions),
  • Retrains a ResNet-18 on the purified dataset, and
  • Evaluates the final classifier on test data using metrics (ACC, ASR, TPR, FPR).

All module interactions strictly follow the design and configuration parameters provided (and read directly from config.yaml). This ensures experimental reproducibility and strict alignment with the VDC method as described in the paper.
